# HWP/HWPX 링크 드래그앤드롭 구현 계획

## 1. 목표

브라우저에서 드래그된 HWP/HWPX 링크를 앱이 식별하고,
임시 다운로드 후 기존 문서 열기 경로로 연결하는 1차 기능을 안전하게 구현한다.

이번 단계는 다음 세 가지를 핵심 축으로 한다.

1. 드래그 데이터 전체 분석
2. 직접 파일 링크 우선 처리
3. 응답 헤더 기반 파일 판별

## 2. 설계 원칙

### 원칙 A. 기존 로컬 파일 열기 경로 보존

링크 기능은 기존 파일 드롭/파일 열기 로직 위에 추가한다.
기존 성공 경로는 건드리지 않고 재사용한다.

### 원칙 B. 판별 실패는 안전 실패

링크를 확신할 수 없으면 문서를 열지 않는다.
억지로 시도하지 않고 사용자 안내 후 종료한다.

### 원칙 C. 단계별 커밋

작업은 작은 단위로 나누어 진행한다.

각 단계마다:

- 구현
- `npx tsc --noEmit`
- 필요 시 `cargo check`
- 수동 확인
- 스테이징/커밋

## 3. 대상 구조

예상 수정 지점:

- 프런트 드롭 이벤트 처리부
- Tauri 백엔드 다운로드/임시 파일 저장 커맨드
- 기존 문서 열기 브리지
- 사용자 안내(UI toast / 상태 메시지)

예상 신규 모듈:

- `link-drop` 또는 `remote-open` 계열 유틸
- URL 판별기
- 헤더 검사기
- 임시 다운로드 저장기

## 4. 구현 단계

## Step 1. 드롭 데이터 표준화

### 목표

브라우저마다 다른 드롭 포맷을 하나의 해석 모델로 정리한다.

### 작업

- 드롭 이벤트에서 아래 포맷 수집
  - `File`
  - `DownloadURL`
  - `text/uri-list`
  - `text/plain`
- 우선순위 규칙 정의
- 최종적으로 `DroppedResourceCandidate[]` 같은 내부 구조로 정규화

### 산출물 예시

```ts
type DroppedResourceCandidate = {
  kind: 'file' | 'url';
  source: 'file' | 'download-url' | 'uri-list' | 'plain-text';
  name?: string;
  url?: string;
  file?: File;
};
```

### 검증

- 기존 로컬 파일 드롭이 그대로 열리는지
- 브라우저에서 텍스트 링크 드롭 시 URL 문자열이 잡히는지

## Step 2. URL 1차 판별기

### 목표

직접 파일 링크를 가장 먼저 빠르게 처리한다.

### 작업

- URL 스킴 검사
  - `http`, `https`만 허용
- URL path의 확장자 검사
  - `.hwp`
  - `.hwpx`
- 직접 파일 링크면 바로 다운로드 후보로 분류

### 검증

- 직접 `.hwp/.hwpx` 링크 드롭 시 다운로드 시도까지 가는지
- 일반 웹페이지는 직접 링크로 오판하지 않는지

## Step 3. 헤더 기반 파일 판별기

### 목표

확장자가 없는 다운로드 링크도 지원한다.

### 작업

- 우선 `HEAD` 요청 시도
- 실패하거나 정보가 부족하면 제한된 `GET` 요청 fallback
- 아래 헤더 확인
  - `Content-Type`
  - `Content-Disposition`
- 판별 규칙:
  - MIME 타입이 HWP/HWPX 관련이면 통과
  - 파일명이 `.hwp/.hwpx`면 통과
  - `octet-stream`이면 파일명 기반 추가 판단

### 검증

- 확장자 없는 실제 다운로드 링크가 열리는지
- 일반 HTML 페이지를 HWP로 오판하지 않는지

## Step 4. 임시 다운로드 및 저장

### 목표

원격 파일을 안전한 임시 경로에 저장한다.

### 작업

- 백엔드 Tauri command 추가
- 임시 디렉터리 생성
- 안전한 파일명 sanitize
- 다운로드 저장
- 저장 결과로 최종 로컬 경로 반환

### 검증

- 다운로드된 파일이 실제 임시 폴더에 생성되는지
- 실패 시 깨진 파일이 남지 않는지

## Step 5. 기존 문서 열기 경로 연결

### 목표

다운로드된 임시 파일을 기존 문서 열기 흐름으로 연동한다.

### 작업

- 이미 사용하는 문서 로딩 함수를 재사용
- 새 링크 기능은 “파일 경로 제공자” 역할만 수행
- 성공 시 기존 파일 열기와 같은 UI 결과를 내야 함

### 검증

- 드롭된 링크 문서가 기존 로컬 파일처럼 열리는지
- 제품 정보, 메뉴, 편집 상태 등 다른 기능이 깨지지 않는지

## Step 6. UX 및 실패 안내

### 목표

실패 원인을 사용자가 이해할 수 있게 만든다.

### 작업

- 상태 메시지/토스트 추가
- 실패 유형 구분
  - 지원되지 않는 링크
  - 다운로드 실패
  - 파일 판별 실패
  - 문서 열기 실패

### 검증

- 실패 시 에디터가 멈추지 않는지
- 사용자가 “왜 안 열렸는지” 알 수 있는지

## 5. 기술 판단 상세

## 5.1 헤더 판별 구현 난이도

난이도는 중간 수준으로 본다.

이유:

- HTML 파싱보다 훨씬 단순
- 판별 기준이 명확
- 리다이렉트 처리와 `HEAD/GET` fallback 정도만 추가하면 됨

주의:

- 일부 서버는 `HEAD`를 막는다
- 일부 서버는 MIME 타입을 부정확하게 보낸다
- 따라서 `Content-Disposition` 병행이 중요하다

## 5.2 위험 요소

### 위험 1. 오탐지

일반 바이너리 파일을 HWP로 오인할 수 있다.

대응:

- MIME + 파일명 병행
- 확신이 없으면 실패 처리

### 위험 2. 로그인/세션 링크

브라우저 세션을 앱이 공유하지 못하면 다운로드 실패 가능

대응:

- 1차 버전은 실패 안내만 제공

### 위험 3. 가비지 임시 파일

실패 중간 생성 파일이 남을 수 있다.

대응:

- 다운로드 실패 시 즉시 삭제
- 성공 후 문서 열기 실패 시도 정리

## 6. 로그 및 관측성

초기 구현에는 아래 로그를 남긴다.

- 드롭 이벤트 포맷별 감지 결과
- 최종 선택된 URL
- 직접 링크 여부
- 헤더 판별 결과
- 저장된 임시 파일 경로
- 문서 열기 성공/실패

로그 목적:

- 브라우저별 드래그 포맷 차이 확인
- 지원되지 않는 사례 수집
- 이후 2차 개선 근거 확보

## 7. 테스트 시나리오

### 정상

1. 로컬 `.hwp` 파일 드롭
2. 로컬 `.hwpx` 파일 드롭
3. 직접 `.hwp` URL 드롭
4. 직접 `.hwpx` URL 드롭
5. 확장자 없지만 헤더상 `.hwp`인 URL 드롭

### 실패

1. 일반 검색 결과 페이지 링크 드롭
2. HTML 문서 링크 드롭
3. 404 링크 드롭
4. 권한 필요한 링크 드롭
5. `blob:` 링크 드롭

## 8. 승인 후 실제 작업 순서

실제 구현은 아래 순서로 진행한다.

1. 드롭 데이터 표준화
2. 직접 URL 판별
3. 헤더 판별
4. 임시 다운로드 저장
5. 기존 문서 열기 연결
6. UX/오류 안내
7. 최종 회귀 테스트

## 9. 승인 체크포인트

구현 시작 전 아래를 승인받는다.

1. 1차 범위는 HTML 파싱 없이 `직접 URL + 헤더 판별`까지만
2. 로그인/세션 링크는 1차에서 미지원
3. 실패 시 열지 않고 안내하는 보수적 정책 유지
4. 임시 다운로드는 앱 전용 temp 경로 사용

## 10. 승인 후 버전 관리 방식

승인 후 작업이 시작되면:

- 단계별 커밋
- 버전 증가
- 릴리스 노트 반영

으로 진행한다.

즉, 이번 문서는 구현 시작 전 기준 설계 문서이며,
실제 코드는 승인 이후에만 추가한다.
