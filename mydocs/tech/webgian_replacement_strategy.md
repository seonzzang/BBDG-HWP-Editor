# 한컴 웹기안기 대체 소프트웨어 개발 전략

## 1. 배경

### 1.1 시장 현황
- 한컴 웹기안기(WebGian)가 대부분의 공공기관에서 도입/운영 중
- 독점적 지위로 인해 고객 피드백 수용이 거의 없는 상태
- 2026년 상반기 가격 인상 통지 (연 구독 방식, 연당 3,500만원 이상)

### 1.2 고객 Pain Point
| 항목 | 내용 |
|------|------|
| 성능 | JS 기반 아키텍처의 구조적 한계로 느린 속도에 대한 불만 빈번 |
| 피드백 | 고객 요구사항/개선 피드백을 거의 수용하지 않음 |
| 비용 | 고가 구독료 + 지속적 인상, 대안 부재로 인한 종속 |

### 1.3 시장 경쟁 분석 (HWP 5.0 바이너리 포맷 기준)

**주의**: HWPX(XML 기반)는 구현 사례가 다수 존재하나, HWP 5.0 바이너리(.hwp)는 상황이 다르다.

**상업 제품 — "읽기+편집+저장+렌더링" 완전 구현**:
| 제품 | 개발사 | 비고 |
|------|--------|------|
| 한글/웹기안기 | 한글과컴퓨터 | 원본 개발사. 서버 의존 OT 아키텍처 |
| Polaris Office | 폴라리스오피스(구 인프라웨어) | 한컴 SDK 미사용 독자 엔진. 서버리스 WASM 방식 브라우저 편집. 모바일/웹/데스크톱 지원. LG인화원, 경남교육청 등 B2B 웹오피스 확장 중. 유일한 상업 경쟁 사례 |

**상업 제품 — 부분 구현 (뷰어/변환, 편집 제한적)**:
| 제품 | 개발사 | 비고 |
|------|--------|------|
| 사이냅 문서뷰어/에디터 | 사이냅소프트 | HWP 96~2024 뷰어, 문서 변환 서버, Document AI. 공공기관 뷰어 시장 점유. HWP 바이너리 프로그래매틱 생성 API 미제공. 서버 기반 SaaS 구조 |

**오픈소스 — 부분 구현만 존재 (렌더링/조판 없음)**:
| 프로젝트 | 언어 | 읽기 | 쓰기 | 렌더링 | 라이선스 |
|----------|------|------|------|--------|----------|
| hwplib (neolord0) | Java | O | O | X | Apache 2.0 |
| libhwp (accforaus) | Kotlin | O | O | X | - |
| pyhwp (mete0r) | Python | O | X | X | AGPL v3 |
| hwp.js (hahnlee) | JS | O | X | HTML ~20% | MIT |
| hwpers (Indosaram) | Rust | O | 주장 | SVG 주장 | - |
| hwp-rs (hahnlee) | Rust | O | X | X | Apache 2.0 |

**LibreOffice**: 내장 hwpfilter는 **HWP 3.0까지만 지원**, HWP 5.0은 서드파티 확장(H2Orestart)으로 ODT 변환 후 읽기만 가능

**핵심 결론**:
- HWP 5.0 바이너리를 "읽기+편집+저장+렌더링(조판)" 모두 구현한 것은 **한컴과 Polaris Office 단 2곳**
- 사이냅소프트는 뷰어/변환에 강하나 HWP 바이너리 생성 API는 미제공
- 오픈소스는 구조 파싱/텍스트 추출 수준이며, **편집+재조판+렌더링 구현체는 전무**
- 본 제품은 Polaris Office에 이어 **세 번째 완전 구현체**
- **WebAssembly 기반 클라이언트 전용(서버 불필요) HWP 저장은 세계 최초** — 한컴은 서버 의존, Polaris는 WASM 편집은 가능하나 AI 에이전트용 프로그래매틱 생성 API는 없음
- **AI Agent용 HWP 생성 도구는 전무** (상용/오픈소스 모두) — 유일한 포지션
- 이는 높은 기술 진입 장벽이자 동시에 경쟁 우위의 원천
- 상세 비전 및 로드맵은 [project_vision.md](mydocs/tech/project_vision.md) 참조

### 1.5 최종 비전 — AI 에이전트 HWP 보고서 생성

Rust로 처음부터 클린룸 설계를 선택한 근본적 이유:

**"AI 에이전트가 HWP 보고서를 자동으로 생성할 수 있게 하는 것이 최종 목표"**

| 단계 | 목표 | 상태 |
|------|------|------|
| 1단계 | Rust HWP 코어 엔진 (파싱/렌더링/편집/저장) | 진행 중 |
| 2단계 | WebAssembly 기반 웹 편집기 (한컴 대체) | 진행 중 |
| 3단계 | AI 에이전트용 HWP 문서 생성 API | **기반기술 확보** |

**3단계 기반기술 현황**:
- HTML 표 → HWP 표 변환: OLE 2.0 클립보드 pasteHtml로 **구현 완료**
- HWP 바이너리 저장: 수정된 문서를 .hwp로 직렬화 **구현 완료**
- 즉, AI 에이전트가 RAG 기반 전문 보고서를 HTML로 작성 → rhwp MCP 도구로 HWP 변환/저장하는 파이프라인이 **기술적으로 실현 가능한 상태**
- 기존 AI 보고서 워크플로(HTML → PDF)에 더해 **HTML → HWP** 경로가 추가됨

**왜 Rust 클린룸인가?**
- **프로그래밍 가능한 엔진**: AI 에이전트가 API를 통해 HWP 문서를 직접 생성/편집할 수 있는 라이브러리 형태
- **서버 불필요**: WASM 기반으로 클라이언트/엣지에서 독립 실행 → AI 에이전트의 경량 도구로 활용
- **한컴 의존성 제거**: 한컴 SDK/서버 없이 HWP를 생성하므로 AI 워크플로에 자유롭게 통합
- **공공기관 보고서 자동화**: 정형화된 HWP 보고서 양식을 AI가 자동 작성하는 시장이 열림

**시장 기회**:
- 공공기관은 HWP 포맷 보고서를 의무적으로 생산 → AI 보고서 자동화 수요 잠재
- 현재 AI가 HWP를 직접 생성할 수 있는 도구는 **전무** (한컴 SDK도 서버 의존적)
- Rust 라이브러리 + WASM = AI 에이전트가 호출 가능한 유일한 HWP 생성 도구

**AI 보고서 생성 파이프라인 (현재 실현 가능)**:
```
AI Agent (Claude/GPT)
    │  RAG 기반 전문 보고서 작성
    ▼
HTML 보고서 (표, 서식 포함)
    │
    ├── 기존 경로: html_to_pdf() → PDF 파일
    │
    └── 신규 경로: rhwp MCP Tool → pasteHtml() → HWP 저장 → .hwp 파일
```

### 1.6 렌더링 품질 비교 검증 (2026-02-16)

한컴 HWP 설치 시 제공되는 기본 예제 파일 `BookReview.hwp`로 국내 주요 상용 웹 뷰어 렌더링 품질을 비교 검증하였다.

**검증 대상**: 2페이지 — 쪽 기준 텍스트박스 다수 배치 + 문단 텍스트 플로우 + 바탕쪽(Master Page) 글상자

| 제품 | 결과 | 비고 |
|------|------|------|
| 한컴 HWP (데스크톱) | 정상 | 원본 기준 |
| **rhwp (본 제품)** | **정상** | 한컴 HWP와 동일한 결과 (SVG/Canvas 모두) |
| 한컴 Docs (웹) | 처리 실패 | 텍스트박스+문단 플로우 미처리 |
| 폴라리스오피스 (웹) | 처리 실패 | 텍스트박스+문단 플로우 미처리 |

한컴 HWP 기본 예제 `KTX.hwp`로 추가 비교 검증하였다.

**검증 대상**: 가로형 1페이지 — 그라디언트/화살표 마커/도형 다수 배치 + 정교한 레이아웃

| 제품 | 품질 | 비고 |
|------|------|------|
| 한컴 HWP (데스크톱) | 100% | 원본 기준 |
| **rhwp (본 제품)** | **95%** | |
| 폴라리스오피스 (웹) | 90% | |
| 네이버 (웹) | 80% | |
| 사이냅 (웹) | 70% | |

**핵심 시사점**:
- 한컴이 직접 제공하는 기본 예제조차 국내 주요 상용 웹 뷰어 3곳이 모두 처리하지 못함
- 본 제품은 한컴 HWP 데스크톱과 동일하거나 가장 근접한 렌더링 결과를 달성
- 텍스트박스+문단 플로우, 도형+그라디언트 등 정교한 렌더링은 높은 기술 진입 장벽에 해당

**개발 방식의 근본적 차이**:
- 경쟁사도 AI를 활용하면 동일한 수준의 정교한 렌더링을 구현할 수 있음
- 그러나 AI가 정확하게 동작하려면 **HWP 스펙 기술문서, 프롬프트 체계, 코드베이스 컨텍스트**의 축적이 필수
- 본 제품의 진정한 경쟁 해자(moat)는 AI 도구 자체가 아니라, 95개 타스크를 거치며 축적된 **프롬프트 엔지니어링 노하우와 기술문서 자산**
- 이 축적 없이 AI만 도입해도 HWP 텍스트 플로우 같은 정교한 렌더링은 달성하기 어려움
- 결과: 고객 피드백 대응 속도와 효율성에서 압도적 우위 확보

## 2. 제품 포지셔닝

- **고객 대응** — 적극적 피드백 수용 및 맞춤 대응

## 3. 상호운용성 전략

### 3.1 목표
기존 한컴 웹기안기를 도입한 고객사가 **기존 개발된 JS 스크립트를 거의 변경하지 않고** 대체할 수 있도록 API 호환성 확보

### 3.2 API 호환 범위
- 기능정의서(task_43_feature_def.md)에서 한컴 웹기안기 449개 API 분석 완료
- 312개 Action, 137개 SetID/Property 식별
- 7개 카테고리: 문서관리, 보기, 편집환경, 커서/편집, 서식, 개체, 도구

### 3.3 호환 계층 구조
```
고객사 기존 JS 스크립트
        │
        ▼
┌─────────────────────┐
│  StudioExtensionAPI  │  ← 한컴 웹기안기 API 호환 인터페이스
│  (Compatibility Layer)│
├─────────────────────┤
│  CommandDispatcher    │  ← 내부 커맨드 시스템
├─────────────────────┤
│  WASM Bridge         │  ← TypeScript ↔ Rust 통신
├─────────────────────┤
│  Rust Core Engine    │  ← HWP 파싱/렌더링/편집 엔진
└─────────────────────┘
```

### 3.4 교체 시나리오
1. 고객사 웹 페이지에서 한컴 웹기안기 스크립트 참조를 rhwp-studio로 변경
2. 기존 JS API 호출 코드는 StudioExtensionAPI 호환 계층이 처리
3. 필요 시 미지원 API에 대한 매핑/폴백 제공

## 4. 저작권/법적 대응

### 4.1 보호되는 영역 (우리의 독자성)
| 항목 | 상태 | 근거 |
|------|------|------|
| 소스코드 | 완전 독자 구현 | Rust로 처음부터 작성, 한컴 코드 참조/복사 없음 |
| 엔진 아키텍처 | 독자 설계 | Rust+WASM 기반, 한컴과 완전히 다른 구조 |
| 렌더링 방식 | 독자 구현 | Canvas 기반 (한컴은 DOM 기반) |

### 4.2 저작권 비해당 영역
| 항목 | 근거 |
|------|------|
| HWP 파일 포맷 | 한글과컴퓨터 공식 공개 스펙(hwp_spec_5.0) 기반 파싱 |
| API 인터페이스 | API는 아이디어/기능적 요소이며 저작권 보호 대상 아님 |
| 기능적 UI/UX | 메뉴 구조, 단축키, 기능 배치는 기능적 요소로 저작권 비해당 |
| 상호운용성 | 상호운용성 확보 목적의 인터페이스 호환은 저작권 예외 |

### 4.3 판례 근거
- **Oracle v. Google (2021, 미국 연방대법원)**: API 인터페이스의 재구현은 공정 사용(Fair Use)으로 인정. Java API 선언부를 Android에서 재구현한 것이 저작권 침해가 아님을 판결.
- **EU 소프트웨어 지침 (2009/24/EC)**: 상호운용성 확보를 위한 역공학 및 인터페이스 호환 허용
- **한국 저작권법 제101조의4**: 프로그램의 상호운용에 필요한 정보를 얻기 위한 행위는 저작권 침해에 해당하지 않음

### 4.4 HWP 공개 스펙 저작권 조항 (hwp_spec_5.0)

한컴이 공식 공개한 「ᄒᆞᆫ글 문서 파일 구조 5.0」 스펙 문서의 저작권 조항:

**허용**: 누구나 열람, 복사, 배포, 게재, 사용을 자유롭게 할 수 있음

**결과물 저작권**: "본 문서를 참고하여 개발한 결과물에 대한 모든 저작권은 결과물을 개발한 개인 또는 단체에 있을 것"

**필수 기재 의무**: 제품 내 사용자 인터페이스, 매뉴얼, 도움말 및 소스에 다음 문구를 반드시 기재:
> "본 제품은 한글과컴퓨터의 ᄒᆞᆫ글 문서 파일(.hwp) 공개 문서를 참고하여 개발하였습니다."

**제한**: 스펙 기반 결과물로 한컴을 상대로 독점적/배타적 권리를 행사하려는 경우 한컴이 적극 권리행사 가능

### 4.5 부정경쟁방지법 대응 (핵심)

저작권보다 더 까다로운 리스크. 한컴이 공들여 만든 API 생태계와 사용자 경험을 무단으로 '무임승차(Free-riding)'하여 경제적 이익을 침해했다고 주장할 수 있다.

**한컴의 예상 공격 논리**:
- 부정경쟁방지법 제2조 제1호 차목(일반조항): "타인의 상당한 투자나 노력으로 만들어진 성과를 공정한 상거래 관행이나 경쟁질서에 반하는 방법으로 자신의 영업을 위하여 무단으로 사용"
- 한컴의 API 설계/생태계가 "상당한 투자와 노력의 성과"에 해당한다는 주장

**방어 논리 — "독자 엔진 + 호환 어댑터" 구조**:
1. **핵심 제품은 독자적인 HWP 편집 엔진**: Rust+WASM으로 처음부터 구축한 고유 기술로, 한컴과 완전히 다른 아키텍처
2. **호환 계층은 부가적 편의 기능**: 고객의 기존 코드 자산 보호를 위해 '레거시 코드 호환용 어댑터'를 부가적으로 제공하는 것
3. **상호운용성 목적의 정당성**: 고객의 벤더 종속(Vendor Lock-in) 해소라는 정당한 경쟁 목적
4. **API는 인터페이스일 뿐**: 기능적 호환을 위한 최소한의 인터페이스 매핑이며, 한컴의 내부 구현을 복제한 것이 아님

**제품 구조적 방어**:
```
┌──────────────────────────────────┐
│   독자 제품 (핵심 가치)           │
│   • Rust HWP 엔진 (독자 구현)     │
│   • WASM Canvas 렌더링 (독자)     │
│   • 자체 UI/UX (독자 디자인)      │
├──────────────────────────────────┤
│   호환 어댑터 (부가 편의)          │  ← 분리 가능한 선택적 계층
│   • StudioExtensionAPI           │
│   • 기존 JS 스크립트 호환 매핑     │
└──────────────────────────────────┘
```

**실행 원칙**:
- 호환 어댑터는 코어 엔진과 분리된 별도 패키지로 구현 (물리적 분리)
- 마케팅에서 "한컴 대체"가 아닌 "HWP 문서 편집기"로 포지셔닝
- 호환 어댑터는 "고객 편의를 위한 마이그레이션 지원" 용도로 설명

**호환 어댑터 구현 전략**:

rhwp 코어는 독자적 API를 설계하고, 호환 어댑터는 별도 패키지로 분리하여 어댑터 패턴으로 위임한다.

```
[rhwp 독자 API]              ← 자체 설계, 독립 구현
    ↑                           exportHwp(), loadDocument() 등
[호환 어댑터 레이어]            ← 별도 패키지 (rhwp-compat-hwpctrl/)
    ↑                           어댑터 패턴: 시그니처 변환 + 위임
[HwpCtrl 호환 인터페이스]      ← 기존 고객 JS 스크립트가 호출
                                 SaveAs(), SaveDocument() 등 67개 메서드
```

- **물리적 분리**: `rhwp-compat-hwpctrl/` 별도 패키지로 구현. rhwp-studio 코어에 포함하지 않음
- **어댑터 패턴**: HwpCtrl 메서드 시그니처를 받아 내부적으로 rhwp 독자 API를 호출하여 위임
  - 예: `SaveAs(fileName, format)` → `wasm.exportHwp()` + Blob 다운로드
  - 예: `SetCharShape(key, value)` → `wasm.applyCharFormat()`
- **선택적 적용**: 고객이 기존 JS 스크립트 호환이 필요한 경우에만 어댑터 패키지를 추가 로드
- **일괄 구현**: 67개 HwpCtrl 메서드를 한번에 구현하되, 모든 내부 로직은 rhwp 독자 API 호출로 처리
- **문서화**: "기존 HwpCtrl 기반 시스템에서 rhwp로 마이그레이션을 돕기 위한 어댑터"로 명시

이 구조는 "API 호환성 제공(상호운용성)"이지 "API 복제"가 아님을 아키텍처 수준에서 입증한다.
