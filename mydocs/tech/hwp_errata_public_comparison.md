# 오픈소스 진영에서 HWP 쓰기는 왜 아직 사용할 수 없는가

> 작성일: 2026-02-23 (최종 갱신: 2026-03-16)
> 분석 대상: hwplib(Java), pyhwp(Python), openhwp(Rust), python-hwpx(Python), 공개 웹 자료

---

## 요약

한글 문서 형식에는 두 가지가 있다: **HWP 5.0**(바이너리)과 **HWPX**(XML/OPC). 오픈소스 진영은 두 형식 모두에서 **읽기**는 가능하지만, **한컴 한/글에서 정상적으로 열리는 파일을 쓰는 것**은 근본적으로 다른 문제다.

- **HWP 바이너리 쓰기**: 공개 스펙에 기술되지 않은 암묵적 직렬화 규칙이 존재하며, 이를 위반하면 한컴이 파일을 거부한다. 가장 앞선 hwplib(Java)조차 일부 규칙을 지키지 못해 **한컴에서 "손상된 파일"로 처리**된다.
- **HWPX(XML) 쓰기**: 바이너리의 벽을 우회하여 XML 기반으로 접근한 프로젝트가 등장했고, **python-hwpx**가 편집·저장에 성공했다. 그러나 **HWPX는 한/글 2022+ 전용**으로, 대다수의 공공기관·기업이 사용하는 .hwp 파일과는 호환되지 않는다.

본 문서는 **왜 오픈소스에서 HWP 쓰기가 아직 사용할 수 없는 수준인지**, 바이너리와 XML 양쪽의 현황을 실증적으로 분석한다. 현존하는 모든 HWP/HWPX 오픈소스 프로젝트는 **개인 수준에서 개발**된 것이며, 바이너리 쓰기에 필요한 역공학과 체계적 검증을 완수한 사례는 극히 드물다.

## 2. 바이너리의 벽 — "스펙을 따르면 파일이 깨진다"

한컴이 공개한 [HWP 5.0 문서 파일 구조](https://www.hancom.com/etc/hwpDownload.do)는 **레코드 구조와 필드 목록**을 제공한다. 그러나 이 스펙에는 다음이 **빠져 있다**:

### 2.1 문서화되지 않은 직렬화 규칙

| 규칙 | 스펙 기술 | 실제 동작 | 위반 시 결과 |
|------|----------|----------|-------------|
| `char_count` MSB | "글자 수" (표 60) | 리스트 마지막 문단은 `char_count \| 0x80000000` | **파일 구조 인식 실패** |
| 빈 문단 `PARA_TEXT` | 언급 없음 | `char_count=1`이면 `PARA_TEXT` 레코드 생략 필수 | **레코드 파싱 오프셋 어긋남** |
| `control_mask` 재계산 | "컨트롤 마스크" (표 60) | 저장 시 문단 내 컨트롤에서 동적 재계산 필요 | **컨트롤(표/그림) 인식 불가** |
| `control_mask` 인라인 비트 | "컨트롤 마스크" (표 60) | TAB(bit 9), FIELD_END(bit 4), LINE_BREAK(bit 10) 등 인라인 제어문자도 포함 필수 | **한컴 2010 비정상 종료** |
| FIELD_BEGIN/END 순서 | 언급 없음 | FIELD_BEGIN 직후 FIELD_END, trailing END는 해당 컨트롤 직후 배치 | **필드 범위 인식 실패** |
| TAB 확장 데이터 | "크기=8" (표 62) | TAB의 7 code unit 원본 데이터 보존 필수 | **탭 간격 오류** |
| 필드 CTRL_HEADER `memo_index` | field_id까지만 기술 (표 154) | field_id 이후 4바이트(MemoShape 참조) 필수 | **누름틀 안내문 빈 문자열** |

### 2.2 문서화되지 않은 필수 필드

| 필드 | 위치 | 스펙 기술 | 실제 |
|------|------|----------|------|
| `prevent_page_break` | `CommonObjAttr` 끝 | 없음 | 4바이트 필수 — 누락 시 뒤의 필드가 밀림 |
| fill alpha 바이트 | `FillInfo` 끝 | 없음 | 채우기 투명도 — 누락 시 색상 오류 |
| `shadowInfo` | `ShapeComponent` 내부 | 없음 | 그림자 구조체 — 누락 시 이후 데이터 파싱 실패 |
| `memo_index` | 필드 CTRL_HEADER 끝 | 없음 | MemoShape 참조 4바이트 — 누락 시 필드 인식 실패 |

### 2.3 문서화되지 않은 레코드 구조

| 구조 | 스펙 기술 | 실제 |
|------|----------|------|
| 그림 삽입 시 `ShapeComponent` | `" gso"` → `ShapeComponentPicture` | `" gso"` → **`ShapeComponent`(ctrl_id=`"$pic"`)** → `ShapeComponentPicture` |
| 확장 컨트롤 문자 크기 | "8 size" (WCHAR 기준인지 byte 기준인지 모호) | **8 WCHAR = 16 bytes** |

## 3. 실증: openhwp가 실패하는 이유

openhwp는 Rust로 작성된 HWP writer를 포함하지만, 생성된 파일은 한컴에서 열리지 않는다. `body_writer.rs`(1,955줄)를 분석한 결과, **공개 스펙에만 의존하여 작성**되었으며 암묵적 규칙을 하나도 적용하지 않았다.

### 발견된 결함

```
 치명적 ─────────────────────────────────────────────
 
 [§5 위반] char_count MSB 누락
   ▸ body_writer.rs:1817  →  text.len() + 1 (0x80000000 OR 없음)
   ▸ 한컴이 문단 리스트 경계를 인식할 수 없음
 
 [§7 위반] control_mask 항상 0
   ▸ body_writer.rs:1012  →  write_u32(0)
   ▸ 표·그림 컨트롤이 있어도 한컴이 무시함
 
 심각 ───────────────────────────────────────────────
 
 [§15 위반] ShapeComponent 중간 레코드 누락
   ▸ body_writer.rs:1164  →  ctrl_header 직후 ShapeComponentPicture 작성
   ▸ 한컴이 기대하는 ShapeComponent(ctrl_id="$pic") 레코드 없음
 
 [§17 위반] shadowInfo 미작성
   ▸ ShapeComponent에 그림자 정보 구조체 빠짐
   ▸ 이후 데이터의 바이트 오프셋이 모두 어긋남
 
 [§13 위반] prevent_page_break 하드코딩
   ▸ body_writer.rs:1100  →  write_u32(0)
```

### 근본 원인

openhwp의 writer는 **공개 스펙(L1)만을 참조**하여 작성되었다. 스펙에 기술된 필드를 순서대로 빠짐없이 작성했지만, **스펙에 기술되지 않은 규칙**이 적용되지 않아 한컴이 파일을 거부한다.

이는 openhwp의 코드 품질 문제가 아니다. **공개 스펙의 구조적 불완전성** 문제다.

## 4. HWPX 우회 전략 — 바이너리를 포기하고 XML로

HWP 바이너리의 벽을 넘지 못한 오픈소스 진영은 **HWPX(XML 기반)로 우회**하는 전략을 택했다. 그 대표 사례가 **python-hwpx**다.

### 4.1 python-hwpx의 접근

| 항목 | 내용 |
|------|------|
| **형식** | HWPX (KS X 6101:2024, OPC/ZIP + XML) |
| **전략** | 한컴이 만든 Skeleton.hwpx 템플릿을 기반으로 XML 트리를 편집한 뒤 ZIP 재패키징 |
| **기술 스택** | Python + lxml, dataclass 기반 본문 모델 |
| **배포** | PyPI (`pip install python-hwpx`) |

### 4.2 왜 HWPX에서는 쓰기가 가능한가

HWPX가 바이너리 HWP와 근본적으로 다른 점:

| | HWP 5.0 (바이너리) | HWPX (XML) |
|---|---|---|
| **데이터 형식** | CFB 컨테이너 + zlib 압축 + 바이너리 레코드 | ZIP + plain XML |
| **필드 구분** | 고정 바이트 오프셋 — **1바이트라도 밀리면 전체 파일 손상** | XML 태그/속성 — 순서 무관, 누락 시 기본값 적용 |
| **직렬화 규칙** | 암묵적 (§5~§17) — 문서화 안 됨 | XML 스키마(XSD)로 명시적 정의 |
| **관용성** | 엄격 — 바이트 단위 정확성 필수 | 관대 — 한/글이 누락 요소를 자동 복구 |

python-hwpx는 이 관용성 덕분에 **문단, 표, 메모, 머리말/꼬리말** 수준의 편집·저장이 가능하다.

### 4.3 그럼에도 HWPX의 한계

python-hwpx가 성공적인 프로젝트임에도 **.hwp 바이너리를 대체할 수 없는** 이유:

1. **호환 범위가 한/글 2022 이후로 제한**
   - 공공기관·기업의 기존 문서 대부분이 .hwp (바이너리)
   - 한/글 2020 이하 버전에서는 .hwpx 파일을 열 수 없음
   - "hwpx로 보내세요"가 아직 통하지 않는 현실

2. **복잡한 개체 지원 불완전**
   - README에 명시: *"add_shape()/add_control()은 한/글이 요구하는 모든 하위 요소를 생성하지 않으므로, 복잡한 개체를 추가할 때는 편집기에서 열어 검증해 주세요"*
   - 그림, 도형, 수식 등의 삽입은 여전히 불완전

3. **기존 .hwp 파일을 편집할 수 없음**
   - python-hwpx는 .hwpx만 처리 — .hwp → .hwpx 변환은 한/글 프로그램이 필요
   - 결국 .hwp 파일의 **프로그래매틱 편집·저장**은 바이너리 쓰기 없이 불가능

## 5. hwplib도 "손상된 파일"로 처리된다

hwplib(Java)는 역공학을 통해 일부 암묵적 규칙을 발견하고 구현했다:

| 규칙 | hwplib 구현 | 소스 위치 |
|------|-----------|----------|
| `char_count` MSB | `lastInList` boolean 필드 + writer autosetter | `ForParagraph.java`, `ForParaHeader.java` |
| `control_mask` 재계산 | `ControlMask` 전용 클래스 | `ParaHeader.java` |
| `shadowInfo` 파싱/쓰기 | `ShadowInfo` 클래스 + reader/writer | `ShapeComponentNormal.java`, `ForShapeComponent*.java` |

그러나 hwplib이 지키지 않는 직렬화 규칙이 여전히 존재한다. L3 미구현 규칙은 주로 **표·그림·도형·채우기 등 복합 컨트롤**에 관련되므로, 텍스트 위주의 단순 문서를 읽고 일부 편집한 뒤 저장하는 것까지는 동작할 수 있다. 그러나 **복합 컨트롤이 포함된 문서를 생성하거나 편집하면 한컴 한/글에서 "손상된 파일입니다"로 처리**된다.

hwplib에서 구현되지 않은 규칙:

| 미구현 규칙 | 결과 |
|-----------|------|
| §6 빈 문단 `PARA_TEXT` 생략 | 레코드 오프셋 어긋남 |
| §8 채우기 alpha 바이트 | 색상 데이터 손상 |
| §13 `prevent_page_break` 필드 | CommonObjAttr 이후 필드 밀림 |
| §14 `attr` bit 15~19 크기 기준 | 확장 컨트롤 크기 오산 |
| §15 `ShapeComponent` ctrl_id `"$pic"` | 그림 레코드 구조 불일치 |

hwplib의 누름틀(ClickHere) 필드 한계:

| 항목 | hwplib | rhwp |
|------|--------|------|
| command 문자열 라운드트립 | ✅ 읽기/쓰기 보존 | ✅ |
| command 내부 파싱 (Direction/HelpState/Name) | ❌ 미구현 | ✅ `guide_text()`, `memo_text()`, `field_name()` |
| 누름틀 안내문/메모/이름 편집 | ❌ 불가 | ✅ `build_clickhere_command()` + CTRL_DATA name |
| properties bit 15 초기 상태 처리 | ❌ 미구현 | ✅ 문서 로드 시 정규화 |
| FIELD_BEGIN/END 직렬화 순서 | 미확인 (writer에 해당 로직 없음) | ✅ trailing END 인터리빙 |
| control_mask 인라인 비트 (TAB/FIELD_END) | ❌ 확장 컨트롤만 계산 | ✅ 전체 제어문자 포함 |

### 5.1 주목할 관찰 — 역량과 결과의 괴리

hwplib의 실제 프로젝트 규모를 확인하면, 쓰기 미완성이 역량 부족과는 무관함을 알 수 있다:

| 항목 | 사실 |
|------|------|
| **개발 기간** | 첫 커밋 2016.12.23 ~ 현재(2026.02), **10년차 프로젝트** |
| **GitHub 지표** | ⭐568 Stars, 181 Forks, 27 Watchers — HWP 오픈소스 중 압도적 1위 |
| **상용화** | README 원문: *"이 라이브러리를 기반으로 하는 **상용 제품이 개발하여 판매**하고 있습니다"* |
| **유료 기술지원** | *"많은 시간을 초래할 수 있는 기술지원 요청이나 유지보수 계약등은 제 메일로 상의해 주셨으면 합니다"* |
| **writer 코드 규모** | **85개 Java 파일** — reader와 대칭적인 완전한 writer 아키텍처 |
| **autosetter 구현** | `autosetter/` 패키지에 `lastInList`, `ControlMask`, `InstanceID` 등 자동 설정 로직 존재 |
| **컨트롤 writer 범위** | Table, Equation, Footnote, Endnote, Header, Footer, Field, Bookmark, ColumnDefine, 각종 GSO 컨트롤 등 **거의 모든 컨트롤에 대한 writer 파일 존재** |
| **쓰기 문서화** | **없음** — 왜 `lastInList`가 필요한지, `ControlMask` 재계산의 이유 등에 대한 설명이 코드 주석에조차 없음 |

10년간 개발하고, 상용 제품의 기반이 되며, 85개의 writer 파일을 보유하고, autosetter까지 구현한 프로젝트가 §6·§8·§13~§15의 직렬화 규칙을 공개 소스에 반영하지 않고 있다. 특히 README에는 **"만들어진 객체를 파일로 저장 : Rewriting_HWPFile, SimpleEditing_HWPFile"**이라고 쓰기를 기능으로 명시하고 있다.

또한 `sample_hwp/basic/` 폴더에는 **27종의 컨트롤별 참조 HWP 파일**(표, 그림, 수식, 각주미주, 글상자, 다각형, 선-사각형-타원, 차트, OLE, 호-곡선 등)이 구비되어 있으며, `Rewriting_HWPFile.java`에서 이 27개 파일 전부를 `HWPReader.fromFile()` → `HWPWriter.toFile()`로 **읽고 다시 저장하는 회귀 테스트**를 수행하고 있다.

### 5.2 결정적 증거 — hwp2hwpx 변환기

같은 개발자가 운영하는 [hwp2hwpx](https://github.com/neolord0/hwp2hwpx)(HWP→HWPX 변환기)를 분석하면, **이 개발자가 L3 규칙을 알면서 공개하지 않는다**는 것이 확인된다:

| errata 규칙 | hwplib reader | hwp2hwpx 변환기 | hwplib writer (공개) |
|-----------|:---:|:---:|:---:|
| **§17 ShadowInfo** | ✅ 파싱 | ✅ HWPX `<shadow>`로 변환 (12개 GSO 컨트롤 전부) | ❌ 미반영 |
| **§13 prevent_page_break** | ✅ `isPreventPageDivide()` | ✅ `holdAnchorAndSO` XML 속성으로 변환 | ❌ 미반영 |
| **§15 ShapeComponent + Picture** | ✅ `ShapeComponentPicture` 파싱 | ✅ `ForPicture extends ForShapeComponent` 구조 | ❌ 미반영 |
| **§8 alpha** | ✅ 파싱 | ⚠️ 일부 `// todo : alpha ??` 주석 | ❌ 미반영 |

**reader에서 올바르게 파싱하고, 변환기에서 HWPX로 정확히 매핑하는 필드를, writer에서는 직렬화하지 않는다.** 특히 2026년 1월에도 *"hwp의 ShadowInfo 객체가 null일 때 오류 해결"*이라는 PR을 처리했으므로, 이 규칙의 존재를 **현재진행형으로 인지**하고 있다.

85개 writer 파일과 autosetter를 10년간 유지한 개발자가 한컴으로 출력 파일을 열어보지 않았을 리 없다. 그리고 **현재 공개된 hwplib 코드대로 직렬화하면 한컴에서 즉시 "손상된 파일입니다"라는 에러에 직면**한다. 10년간 이 문제를 모를 수는 없으므로:

1. ~~L3 규칙이 그만큼 발견하기 어렵다~~ — reader와 변환기에서 해당 필드를 처리하고 있으므로, 모르는 것이 아님
2. **쓰기 핵심 규칙은 공개 소스에 미반영** — 상용 제품 쪽에서는 이미 해결했으나, 완전한 쓰기 코드를 공개 저장소에 커밋하지 않은 것

hwplib의 사례는 **공개 소스코드만으로는 한컴 호환 HWP 쓰기를 달성할 수 없다**는 사실을 재확인시킨다.

즉, **오픈소스 진영에서 가장 앞선 hwplib조차 공개 소스 기준으로는 한컴 호환 쓰기에 실패**하고 있다. openhwp가 L1(공개 스펙)에서 막혔다면, hwplib은 L2(역공학)까지 도달했으나 **L3(독자 발견 규칙)의 코드를 공개하지 않은 것**이다.

## 6. 공개 웹에 HWP 쓰기 정보가 없는 이유

웹 검색 결과, HWP 바이너리 쓰기의 세부 직렬화 규칙을 다루는 **블로그, 포럼 글, 기술 문서는 일체 발견되지 않았다**. 이는 우연이 아니다:

1. **2013년 libghwp 사건**: HWP 역공학 오픈소스 개발자가 코드 도용 피해를 입고 **3년치 소스와 분석 자료를 모두 삭제** (ZDNet, 2013.12.12)
2. **쓰기까지 도달한 프로젝트가 극소수**: 읽기만으로도 구현 난이도가 높아 대부분의 프로젝트가 파서 단계에서 중단
3. **역공학 지식의 비공유 관행**: hwplib처럼 쓰기를 성공한 프로젝트도 코드만 공개하고 **규칙 자체를 문서화하지 않음**
4. **HWPX 우회 선호**: 바이너리의 벽에 부딪힌 개발자들이 .hwpx 쪽으로 이동하면서 .hwp 쓰기 연구 자체가 정체

## 7. 구조적 장벽 — 개인 프로젝트가 넘기 어려운 이유

현존하는 HWP/HWPX 오픈소스 프로젝트는 **모두 개인 개발자 수준**에서 만들어졌다. 기업이나 조직 단위의 프로젝트는 존재하지 않는다. 이는 우연이 아니라 다음의 구조적 장벽 때문이다.

### 7.1 역공학의 필수성

공개 스펙(L1)만으로는 한컴 호환 쓰기가 불가능하다는 것이 이미 실증되었다(§3 openhwp, §5 hwplib). 나머지 규칙은 **한컴이 만든 실제 .hwp 파일을 바이너리 수준에서 분석**하여 역으로 추론해야 한다.

이 과정은:
- 한컴이 공개하지 않은 내부 규칙을 **바이트 단위로 추적**하는 작업
- 스펙 문서와 실제 바이너리가 다를 때, **어느 쪽이 맞는지 실험으로 판별**하는 작업
- 스펙에 아예 언급되지 않은 필드를 **존재 자체를 발견**해야 하는 작업

### 7.2 모든 컨트롤에 대한 직렬화 테스트

HWP 5.0은 **50종 이상의 컨트롤**(표, 그림, 수식, 도형, 각주, 미주, 머리말, 꼬리말, 하이퍼링크, 책갈피, OLE, 차트, 비디오 등)을 지원한다. 각 컨트롤은 고유한 직렬화 구조를 가지며, 하나라도 틀리면 파일 전체가 손상된다.

한컴 호환 쓰기를 달성하려면:

| 단계 | 작업 | 난이도 |
|------|------|--------|
| 1 | 컨트롤별 **참조 .hwp 파일** 생성 (한컴으로 직접 작성) | 수동 |
| 2 | 참조 파일의 **바이너리를 바이트 단위로 분석** | 역공학 |
| 3 | 분석 결과로 **직렬화 코드 구현** | 구현 |
| 4 | 구현된 코드로 쓴 파일을 한컴으로 열어 **1:1 대조** | 검증 |
| 5 | 모든 컨트롤에 대해 **회귀 테스트 케이스 유지** | 지속 운영 |

이 5단계를 **50종 이상의 컨트롤 × 각 컨트롤의 옵션 조합**에 대해 수행해야 한다. 한 컨트롤의 직렬화를 수정하면 다른 컨트롤에 영향을 줄 수 있으므로, **회귀 테스트 없이는 안정성을 보장할 수 없다**.

### 7.3 왜 개인 수준에서 정체되는가

| 장벽 | 설명 |
|------|------|
| **시간** | 50종+ 컨트롤의 역공학·구현·검증 사이클을 1인이 수행 |
| **도구** | 한컴 한/글 프로그램(유료) + 헥스 에디터 + 자동화 테스트 환경 |
| **지식** | 바이너리 역공학, CFB 컨테이너, zlib 압축, 바이트 오프셋 계산 등 **일반 앱 개발과 이질적인 기술 스택** |
| **동기** | 상업적 수익 모델 부재 — 한컴이 무료 뷰어를 제공하므로 읽기 수요는 이미 충족 |
| **위험** | libghwp 사건(2013)처럼 코드 도용 피해 가능성 |

결국 HWP 바이너리 쓰기는 **읽기 파서 구현과는 차원이 다른 엔지니어링 노력**을 요구하며, 이것이 10년 넘게 오픈소스 진영에서 쓰기가 해결되지 않는 근본 원인이다.

## 8. 전체 구도 — 왜 아직 "쓸 수 있는" 상태가 아닌가

```
             ┌─────────────────────────────────┐
             │        한컴 한/글 생태계          │
             ├─────────────────────────────────┤
             │  .hwp (바이너리)    .hwpx (XML)  │
             │  ↑ 90%+ 점유율     ↑ 신규 문서   │
             └──────┬──────────────────┬───────┘
                    │                  │
        ┌───────────┴────────┐    ┌───┴──────────────┐
        │  바이너리 쓰기      │    │  XML 쓰기         │
        │                    │    │                   │
        │  ❌ openhwp        │    │  ✅ python-hwpx   │
        │     (L1 스펙만→   │    │     (XML 편집→OK) │
        │      파일 손상)    │    │                   │
        │                    │    │  한계:            │
        │  ⚠️ hwplib(Java) │    │  - 한/글 2022+ 전용│
        │     (L2 역공학까지→│    │  - .hwp 편집 불가  │
        │      일부 손상)    │    │  - 복잡 개체 미지원 │
        │                    │    │                   │
        │  ✅ rhwp(Rust)    │    │                   │
        │     (L3+L4 독자    │    │                   │
        │      발견 포함)    │    │                   │
        └────────────────────┘    └───────────────────┘
```

### 결론

현재 오픈소스에서 **기존 .hwp 파일을 프로그래매틱하게 편집하고 한컴에서 정상적으로 열리도록 저장**할 수 있는 프로젝트는 **rhwp(Rust)가 유일**하다.

- **HWPX 접근(python-hwpx)**: XML의 관용성 덕분에 쓰기가 가능하지만, .hwp 처리 불가 + 한/글 2022+ 전용
- **바이너리 역공학(hwplib)**: L2까지 도달했으나 L3 규칙 미준수로 "손상된 파일" 처리. 누름틀 등 필드 컨트롤의 command 내부 파싱·편집 기능도 없음
- **바이너리 스펙 기반(openhwp 등)**: L1만으로는 한컴 호환 쓰기가 **원천적으로 불가능**

rhwp는 정밀 렌더링(페이지네이션, 표 분할, 머리말/꼬리말)과 에디터 기능(텍스트 편집, 누름틀 편집, 필드 데이터 바인딩, 그림 삽입)을 모두 지원하는 **유일한 오픈소스 HWP 뷰어/에디터**이다. 정오표 24건(L3 독자 발견 12건 포함)과 troubleshooting 18건의 축적된 역공학 지식이 이를 뒷받침한다.
