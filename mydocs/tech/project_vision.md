# rhwp 프로젝트 최종 목표

## 비전

**AI Agent가 HWP 문서를 읽고·이해하고·생성하는 End-to-End 파이프라인 제공**

한국 공공·기업 문서 생태계의 사실상 표준인 HWP 포맷을 AI가 완전히 다룰 수 있도록,
**시맨틱 파싱(입력)** 과 **네이티브 생성(출력)** 양쪽을 모두 확보한다.

```
┌──────────────────────────────────────────────────────────────────┐
│                   "The Semantic Breaker"                          │
│                                                                   │
│  ┌─────────────────┐     ┌──────────┐     ┌─────────────────┐   │
│  │  hwp_semantic    │     │  AI Agent │     │  rhwp           │   │
│  │  (Python)        │ ──▶ │  (LLM)   │ ──▶ │  (Rust/WASM)    │   │
│  │                  │     │          │     │                  │   │
│  │  HWP → 시맨틱   │     │  분석    │     │  HTML/API → HWP │   │
│  │  파싱·청킹·검색  │     │  추론    │     │  파싱·렌더링     │   │
│  │                  │     │  생성    │     │  편집·직렬화     │   │
│  └─────────────────┘     └──────────┘     └─────────────────┘   │
│       ▲ 입력(Break)                          ▼ 출력(Forge)       │
│       │                                      │                   │
│       └──────────── .hwp ◄───────────────────┘                   │
└──────────────────────────────────────────────────────────────────┘
```

## 브랜드 아이덴티티: "The Semantic Breaker"

### 컨셉

묠니르(Mjölnir) — 파편화된 AI 데이터를 정교한 HWP 문서로 조판(Typesetting)하는 지능형 도구.
단순히 기존의 것을 부수는 망치가 아니라, 시맨틱 구조를 **분해(Break)** 하고 **단조(Forge)** 하는 양면의 도구이다.

### 로고 구성 요소

| 요소 | 상징 | 대응 기술 |
|------|------|-----------|
| **망치 머리** (Dog-eared 문서) | HWP 파일 완벽 호환 | rhwp: HWP 바이너리 파싱·직렬화 |
| **타격 면** (데이터 블록 그리드) | RAG 데이터, 시맨틱 청크 | hwp_semantic: 3-Level 청킹, CTCR 좌표 참조 |
| **손잡이** (차콜 직선 + 육각형) | Rust 코어 + WASM | rhwp: Rust 네이티브, wasm-pack 빌드 |
| **파편 효과** (표·서식 형태) | AI가 HWP를 생성하는 순간 | 표 페이지 분할, 머리말/꼬리말, 자동번호 |

> CTCR: 
> 좌표 기반 표 셀 검색 알고리즘 (Coordinate-based Table Cell Retrieval)
>
> 자연어 질의에서 추출한 키워드를 사용하여 2차원 표(Table)에서 특정 데이터 셀의 값을 검색하는 알고리즘
> 핵심 수식
>
> ```
> C(x,y) = (Kmax(열헤더 키워드), Kmax(행헤더 키워드))
> ```
>
> - `C(x,y)`: 검색 결과 데이터 셀의 좌표
> - `Kmax(열헤더)`: 열헤더 영역에서 키워드가 발견된 최대 x좌표
> - `Kmax(행헤더)`: 행헤더 영역에서 키워드가 발견된 최대 y좌표

### 브랜드 메시지

> **"AI가 쓴 글을 HWP로 완성하는 유일한 도구"**

활용처: rhwp-studio 로딩 화면, CLI ASCII 아트, MCP 서버 도구 아이콘.

## 배경

### 한국 문서 생태계의 현실
- 공공기관, 대기업, 학교, 군대 등에서 HWP는 사실상 표준
- AI 시대에도 "보고서를 HWP로 제출하세요"라는 요구는 변하지 않음

### AI 문서 생성의 현재 한계
- AI Agent(Claude, GPT 등)는 PDF, DOCX, Markdown 생성 가능
- **HWP 바이너리 포맷 생성은 불가능** — 시장에 해당 기술/제품 없음
- DOCX → HWP 변환은 서식 깨짐, 수동 작업 필요

### 기존 솔루션의 한계

**HWP 생태계 기업**:

| 솔루션 | 사업 영역 | AI Agent용 한계 |
|--------|----------|----------------|
| 한컴 webhwp | 웹 기반 HWP 편집 (OT/서버) | 서버 의존, 유료 클라우드, API 비공개, OT 구조 |
| 폴라리스오피스 | 웹/모바일 HWP 편집 (WASM) | B2B 웹오피스 중심, 프로그래매틱 생성 API 없음 |
| 사이냅소프트 | 문서 뷰어/변환/에디터 서버 | 뷰어/변환 특화, HWP 생성 API 미제공, 서버 의존 |

- **폴라리스오피스** (인프라웨어): 서버리스 웹어셈블리 방식으로 브라우저에서 HWP 편집 가능.
  LG인화원, 경남교육청 등 B2B 웹오피스로 확장 중. 그러나 "사람이 UI로 편집하는" 제품이며,
  AI Agent가 코드로 HWP를 생성하는 API/SDK는 제공하지 않음.

- **사이냅소프트**: 사이냅 문서뷰어(HWP 96~2024 지원), 사이냅 에디터, 문서 변환 서버 등
  Document AI 솔루션 제공. 공공기관 문서 뷰어 시장 점유. 그러나 **뷰어/변환** 특화이며
  HWP 바이너리를 프로그래매틱하게 생성하는 API는 없음. 서버 기반 SaaS 구조.

**오픈소스/기타**:

| 솔루션 | 한계 |
|--------|------|
| LibreOffice CLI | HWP 읽기 제한적, 쓰기 불가 |
| Python hwp5 | 읽기 전용, 쓰기 미지원 |
| HWPX (OOXML 기반) | .hwp 바이너리 파일과 호환 불가 |

**공통 한계**: 기존 솔루션은 모두 **"사람이 소프트웨어를 사용하여 문서를 다루는"** 패러다임이다.
AI Agent가 함수 호출 한 번으로 HWP를 생성하는 도구는 존재하지 않는다.

**→ 로컬에서 .hwp 바이너리를 프로그래매틱하게 생성하는 도구는 존재하지 않음 (오픈소스/상용 모두)**

## 전체 생태계: hwp_semantic + rhwp

### 두 프로젝트의 역할

| | hwp_semantic (Python) | rhwp (Rust/WASM) |
|---|---|---|
| **방향** | HWP → AI (입력) | AI → HWP (출력) |
| **핵심 기능** | 시맨틱 파싱, 청킹, 벡터 검색 | 렌더링, 편집, 직렬화, 변환 |
| **AI 연동** | MCP 서버 (11개 Tool) | MCP 서버 (생성/수정 Tool) |
| **배포** | PyPI, Docker (MCP/BatchLoader) | PyPI, npm, Crate, MCP 바이너리 |
| **라이선스** | MIT | MIT |

### hwp_semantic — 입력(Break) 파이프라인

HWP 바이너리를 시맨틱 구조로 분해하여 AI가 이해할 수 있는 형태로 변환한다.

```
HWP/HWPX 문서
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  hwp_semantic                                            │
│                                                          │
│  시맨틱 파싱 ─▶ Markdown/JSON 내보내기                    │
│      │                                                   │
│      ▼                                                   │
│  3-Level 청킹 (Table → Row → Cell)                       │
│      │         Row-Level: "[부서별 운영비] IT팀: 500M"    │
│      │         CTCR: 셀 좌표 기반 정밀 참조               │
│      ▼                                                   │
│  Vector Store (PostgreSQL + pgvector, BGE-M3 1024dim)    │
│      │                                                   │
│      ▼                                                   │
│  MCP 서버 (11개 Tool)                                     │
│      • search_fiscal_data (하이브리드 검색, RRF)          │
│      • get_time_series / compare_periods (Pivot Query)   │
│      • get_document_structure (컨텍스트 윈도우)            │
│      • get_cell_value / get_cell_range (CTCR 좌표)       │
└─────────────────────────────────────────────────────────┘
    │
    ▼
Claude Desktop / 로컬 LLM / 폐쇄망 AI
```

**운용 규모**: 182개 통합재정통계 HWP 문서, 5,500+ 행 청크, 11,600+ 셀 벡터화 완료.
BatchLoader로 Docker 원클릭 일괄 적재. 동적 도메인 탐지, 이미지 pHash 중복감지 지원.

### rhwp — 출력(Forge) 파이프라인

AI가 생성한 콘텐츠를 HWP 바이너리로 변환·저장한다.

```
AI Agent 출력 (HTML / 구조화 API)
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  rhwp                                                    │
│                                                          │
│  HTML → HWP 변환 (1차)                                   │
│      h1~h6, p, table, strong, em → HWP 문단/표/서식      │
│                                                          │
│  네이티브 HWP 생성 API (2차)                              │
│      create_hwp(), add_paragraph(), add_table()          │
│                                                          │
│  기존 HWP 읽기/수정/저장                                  │
│      파싱 → 편집 → surgical_update → 직렬화              │
│                                                          │
│  렌더링 (SVG / Canvas / HTML)                             │
│      페이지네이션, 표 분할, 클리핑, 머리말/꼬리말          │
│                                                          │
│  WASM 웹 편집기 (rhwp-studio)                             │
│      캐럿, 텍스트 입력, 서식 툴바, Ctrl+S 저장            │
└─────────────────────────────────────────────────────────┘
    │
    ▼
.hwp 파일 출력 (한컴오피스 호환)
```

### 완전 사이클: 검색 → 분석 → 생성

두 프로젝트를 결합하면 AI가 HWP 문서를 **읽고 → 이해하고 → 새로 만드는** 완전한 사이클이 성립한다.

```
                    ┌─────────────────────────────────┐
                    │         AI Agent (LLM)           │
                    │                                  │
                    │  "2024년 대비 경상지출 증감      │
                    │   분석 보고서를 HWP로 작성해줘"  │
                    └───────┬─────────────┬────────────┘
                            │             │
                   MCP Tool Call    MCP Tool Call
                            │             │
              ┌─────────────▼──┐    ┌─────▼──────────────┐
              │ hwp_semantic   │    │ rhwp               │
              │ MCP 서버       │    │ MCP 서버            │
              │                │    │                     │
              │ search_fiscal_ │    │ html_to_hwp()       │
              │ data("경상지출")│    │                     │
              │ compare_periods│    │ → .hwp 파일 출력    │
              │ (2023, 2024)   │    │                     │
              └────────────────┘    └─────────────────────┘
```

**시나리오 예시**:
1. AI가 hwp_semantic MCP로 재정 데이터 검색·비교 (`search_fiscal_data`, `compare_periods`)
2. 분석 결과를 HTML 보고서로 구성 (표, 차트 설명, 증감률 분석)
3. rhwp MCP로 HTML → HWP 변환 (`html_to_hwp`)
4. 완성된 .hwp 파일 출력 — 한컴오피스에서 바로 열어 제출 가능

## 최종 목표: AI Agent의 HWP 도구

AI Agent가 HWP 문서를 **로컬에서 직접** 생성/읽기/수정/저장할 수 있는 도구를 제공한다.
기존 HWP 생태계(한컴, 폴라리스, 사이냅)가 모두 "사람이 UI로 조작하는" 패러다임인 반면,
rhwp는 **"AI가 코드로 조작하는"** 패러다임을 최초로 실현한다.

### 두 가지 제공 경로

```
경로 1: 라이브러리 직접 호출 (Phase 1)          경로 2: MCP 서버 (Phase 2)

[Python/Node.js 스크립트]                      [Claude Code / Cowork]
    │                                              │
    │  import rhwp                                 │  MCP Tool Call (JSON-RPC)
    │  rhwp.html_to_hwp(html, path)                ▼
    │  rhwp.hwp_to_pdf(hwp, path)              ┌─────────────────────────┐
    ▼                                          │ rhwp MCP Server         │
  .hwp / .pdf 파일 출력                         │                         │
                                               │ html_to_hwp()           │
                                               │ hwp_to_pdf()            │
                                               │ create_hwp()            │
                                               │ read_hwp()              │
                                               │ modify_hwp()            │
                                               │ merge_hwp()             │
                                               └─────────────────────────┘
                                                   │
                                                   ▼
                                                 .hwp / .pdf 파일 출력
```

**경로 1 (라이브러리)**: `pip install rhwp` 또는 `npm install rhwp`로 설치하여
기존 스크립트/스킬에서 `html_to_hwp()` 한 줄로 HWP를 생성한다. 최소 통합 비용.

**경로 2 (MCP 서버)**: AI Agent가 읽기/수정/템플릿 채우기 등 고급 문서 조작을 수행한다.
MCP 프로토콜(JSON-RPC over stdio)로 Claude Code/Cowork에 직접 연동.

### 기존 생태계와의 차별점

| | 한컴/폴라리스/사이냅 | rhwp |
|---|---|---|
| **사용 주체** | 사람 (UI 클릭/타이핑) | AI Agent (함수 호출) |
| **접근 방식** | 웹 에디터, 뷰어, 변환 서버 | 라이브러리/MCP 서버 |
| **서버 의존** | 필수 (SaaS/클라우드) | 없음 (로컬 처리) |
| **HWP 생성 API** | 미제공 | 핵심 기능 |
| **AI 통합** | 별도 개발 필요 | MCP/스킬 문서로 즉시 연동 |

### 사용 시나리오

1. **보고서 생성**: "이번 분기 매출 데이터로 보고서를 HWP로 만들어줘"
2. **공문서 작성**: "기안문 양식에 맞춰 출장 신청서 HWP 작성해줘"
3. **문서 변환**: "이 Markdown 보고서를 HWP로 변환해줘"
4. **기존 문서 수정**: "이 HWP 파일의 3페이지 표에 데이터 추가해줘"
5. **일괄 처리**: "폴더 내 모든 HWP 파일에서 텍스트 추출해줘"
6. **재정 데이터 분석 보고서**: "2023년 대비 2024년 경상지출 증감 분석 보고서를 HWP로" *(hwp_semantic 연동)*

### HWP 생성 2단계 전략

#### 1차: HTML → HWP 변환 (빠른 시장 진입)

AI 모델은 이미 HTML 출력에 능숙하다. 표, 제목, 서식을 자연스럽게 생성한다.
이 HTML을 HWP로 변환하는 것만으로 즉시 가치를 제공할 수 있다.

```
AI Agent ──▶ HTML 생성 ──▶ rhwp html_to_hwp() ──▶ .hwp 출력

지원 태그:
  h1~h6  → HWP 개요 수준 (문단 모양)
  p      → HWP 일반 문단
  table  → HWP 표 (셀 병합, 테두리, 배경색)
  strong → HWP 굵게 (글자 모양)
  em     → HWP 기울임 (글자 모양)
  ul/ol  → HWP 글머리표/번호 목록
  img    → HWP 이미지 삽입 (BinData)
```

**장점**: AI 모델 수정 불필요, 기존 html_to_pdf() 워크플로우에 한 줄 추가로 HWP 출력 지원.

#### 2차: 네이티브 HWP 생성 API (고품질)

HTML 중간 단계를 건너뛰고 HWP 고유 서식을 직접 제어한다.

```
AI Agent ──▶ rhwp 구조화 API ──▶ .hwp 출력

API:
  create_hwp()                      빈 문서 생성
  add_paragraph(text, style)        문단 추가 (개요 수준, 글자/문단 모양)
  add_table(rows, cols, data)       표 추가 (셀 병합, HWPUNIT 크기 제어)
  set_page_def(paper, margins)      용지 설정 (A4, B5, 여백)
  add_header_footer(text, pos)      머리말/꼬리말
  add_page_number(format, pos)      쪽 번호 (아라비아/로마/원문자)
  save(path)                        HWP 바이너리 직렬화
```

**장점**: HWPUNIT 기반 정밀 여백, 문단 모양/글자 모양 직접 제어, 머리말·꼬리말·쪽 번호 등 HWP 고유 기능 활용.

#### 단계별 가치 비교

| | 1차 (HTML→HWP) | 2차 (네이티브) |
|---|---|---|
| **구현 난이도** | 중 (HTML 파싱 + 매핑) | 고 (전체 문서 모델 구축) |
| **시장 진입 속도** | 빠름 (기존 HTML 워크플로우 활용) | 느림 (API 설계·문서화 필요) |
| **서식 정밀도** | HTML 수준 (제한적) | HWP 네이티브 수준 (완전) |
| **AI 통합 비용** | 최소 (한 줄 추가) | 중 (새 API 학습 필요) |
| **기존 자산 활용** | pasteHtml 파이프라인 확장 | 파싱/직렬화 엔진 전체 활용 |

### 템플릿 기반 문서 자동화

공공기관/기업에서 사용하는 정형화된 HWP 템플릿을 활용한 문서 자동 생성.

**워크플로우**:
```
1. 기관 템플릿 (.hwp) 등록
   예: 기안문, 출장신청서, 회의록, 업무보고서, 품의서, 공문 등

2. AI Agent가 read_hwp()로 템플릿 구조 파악
   → 표 위치, 입력 필드, 문서 구조 해석

3. 사용자 지시에 따라 내용 자동 채움
   "출장 신청서 작성. 일시: 2/15~17, 목적지: 부산, 목적: 현장 점검"
   → modify_hwp()로 해당 위치에 텍스트 삽입/표 셀 편집

4. 완성된 HWP 저장 → 바로 제출 가능
```

**기존 방식과 비교**:
```
사람:  한컴오피스 실행 → 양식 파일 열기 → 수동 타이핑 → 저장 → 결재
AI:    "출장 신청서 써줘" → 자동 완성된 HWP 출력 → 결재
```

**템플릿 활용의 장점**:
- 기관별 고유 서식(머리글, 관인, 문서번호 체계)이 그대로 유지됨
- AI가 새 문서를 만드는 것보다 기존 양식을 채우는 것이 훨씬 정확
- 이미 검증된 읽기→수정→저장 파이프라인으로 구현 가능

### 템플릿 매니페스트 (프롬프트 기반 정밀 제어)

각 템플릿에 **사용법 프롬프트**를 함께 제공하여 AI가 문서 구조를 분석하지 않고도
정확한 위치에 정확한 형식으로 데이터를 입력할 수 있도록 한다.

```yaml
# templates/출장신청서.yaml
name: 출장신청서
file: 출장신청서.hwp
fields:
  - name: 성명
    location: { table: 0, row: 0, col: 1 }
    type: text
  - name: 출장기간
    location: { table: 0, row: 1, col: 1 }
    type: date_range
    format: "yyyy.mm.dd ~ yyyy.mm.dd"
  - name: 목적지
    location: { table: 0, row: 2, col: 1 }
    type: text
  - name: 출장목적
    location: { table: 0, row: 3, col: 1, colspan: 6 }
    type: multiline
prompt: |
  출장신청서를 작성합니다.
  사용자가 제공한 정보로 각 필드를 채우고,
  출장 목적은 공문서 어조로 작성합니다.
  날짜는 반드시 yyyy.mm.dd 형식, 금액은 천 단위 콤마 포함.
```

**매니페스트의 역할**:
- AI가 `read_hwp()` 없이 바로 정확한 셀 위치에 데이터 삽입 가능
- 데이터 형식(날짜, 금액, 어조)까지 명시하여 일관된 품질 보장
- 템플릿 제작자의 의도를 AI에게 직접 전달
- 기관별 자체 템플릿 라이브러리 구축 가능

## 기술 아키텍처

### webhwp (한컴) vs rhwp 비교

| | webhwp (한컴) | rhwp |
|---|---|---|
| **파싱** | 서버 | Rust 네이티브 / WASM |
| **직렬화** | 서버 (`/handler/convert/`) | Rust 네이티브 |
| **편집 동기화** | OT 엔진 (WebSocket) | 로컬 처리 |
| **렌더링** | Canvas (nwr 브릿지) | SVG / Canvas (WASM) |
| **서버 의존성** | 필수 (세션/메모리 점유) | 없음 |
| **배포 형태** | 클라우드 SaaS | 로컬 바이너리 / MCP 서버 |
| **비용** | 사용자 수 비례 서버 비용 | 0 (정적 호스팅만 필요) |

### AI Agent 도구로서의 근본적 차이

webhwp는 **사람의 실시간 협업 편집**을 위해 설계된 OT 아키텍처이다.
AI Agent가 문서를 생성하는 시나리오와는 근본적으로 맞지 않다.

**webhwp로 AI가 문서를 만들려면**:
```
세션 생성 → WebSocket 연결 → 문서 모델 메모리 로드 →
  텍스트 삽입 Operation 전송 → OT 충돌 해소 (불필요) →
    서식 적용 Operation 전송 → OT 충돌 해소 (불필요) →
      ... (수십 번의 네트워크 왕복) ...
        직렬화 요청 → 서버 변환 → 바이너리 다운로드 → 세션 해제
```

**rhwp로 AI가 문서를 만들면**:
```
rhwp.html_to_hwp(html, "output.hwp")  # 끝. 로컬 함수 호출 1회.
```

| 비교 항목 | webhwp (한컴) | rhwp |
|-----------|-------------|------|
| **AI 호출 방식** | WebSocket 세션 + 다수 Operation | 함수 호출 1회 |
| **네트워크 왕복** | 문서당 수십 회 | 0회 |
| **OT 엔진** | 필수 (AI 혼자 쓸 때도 동작) | 없음 (불필요) |
| **서버 메모리** | 문서당 세션 점유 | 0 (로컬 처리) |
| **동시 생성 한계** | 서버 용량 비례 | 무제한 (각 클라이언트 독립) |
| **한컴의 AI 대응** | OT 우회 경로 신규 개발 필요 | 이미 설계 목적 자체가 AI용 |

**OT의 구조적 한계 — 일괄 편집 불가**:

OT는 각 Operation이 이전 Operation의 결과에 의존하는 순차 처리 구조이다.
여러 편집을 한꺼번에 보내도 서버는 순차적으로 하나씩 OT 변환을 수행해야 한다.

```
webhwp: Op1 → OT변환 → 갱신 → Op2 → OT변환 → 갱신 → ... → OpN → 직렬화
rhwp:   수정1, 수정2, ... 수정N (메모리 직접 조작) → 1회 직렬화
```

한컴이 AI Agent용 헤드리스 서비스를 만들려면 OT를 통째로 우회하여
C/C++ 엔진에서 직접 문서 모델을 조작하는 "배치 모드"를 새로 개발해야 한다.
이는 webhwp 아키텍처의 근본적 변경이며, 사실상 rhwp와 같은 작업을 처음부터 다시 하는 것이다.

**rhwp는 처음부터 "AI가 함수 하나로 HWP를 만든다"를 목표로 설계되었다.**

### rhwp의 구조적 장점

1. **서버 비용 0**: 모든 처리가 클라이언트/로컬에서 완결
2. **오프라인 동작**: 네트워크 없이도 완전한 기능 제공
3. **Rust 네이티브**: MCP 서버로 직접 빌드 가능, 빠르고 안정적
4. **WASM 지원**: 브라우저 기반 AI 도구에서도 활용 가능
5. **단일 바이너리 배포**: 설치/운영 단순

### Phase 1: 최소 실현 (PyPI + 스킬 문서 확장) — 목표: 3월 중순

가장 빠르게 가치를 제공하는 경로. 기존 Claude 스킬 인프라를 그대로 활용.

| 단계 | 작업 | 설명 |
|------|------|------|
| 1 | HTML → HWP 변환 핵심 구현 | HTML 태그를 HWP 문단/표/서식으로 매핑 |
| 2 | HWP → PDF 변환 (hwp2pdf) | 기존 SVG 렌더링 → PDF 변환 파이프라인 |
| 3 | PyO3 Python 바인딩 | `rhwp.html_to_hwp()`, `rhwp.hwp_to_pdf()` API |
| 4 | PyPI 패키지 배포 | `pip install rhwp` |
| 5 | 기존 스킬 문서 확장 | "HWP 출력이 필요하면 `rhwp.html_to_hwp()` 사용" 한 섹션 추가 |

**hwp2pdf 변환 파이프라인**:
```
HWP 파일 → rhwp 파싱 → 페이지 레이아웃 → SVG 렌더링 → PDF 변환
                        (기존 완성)        (기존 완성)    (신규)
```
기존 SVG 렌더링 엔진이 완성되어 있으므로, SVG → PDF 변환 단계만 추가하면 된다.
이 기능은 단독으로도 가치가 크다: 한컴오피스 없이 HWP를 PDF로 변환하는 CLI/라이브러리.

**결과**: AI Agent가 HWP 생성과 PDF 변환을 모두 로컬에서 수행
```
기존 스킬 문서:
  "보고서를 HTML로 생성 → html_to_pdf()로 PDF 변환"
확장:
  "한국어 HWP 포맷 필요 시 → rhwp.html_to_hwp()로 HWP 변환"
  "HWP → PDF 변환 필요 시 → rhwp.hwp_to_pdf()로 PDF 변환"
```

### Phase 2: MCP 서버 (고급 기능) — 목표: 3월 말

읽기/수정/템플릿 등 고급 문서 조작이 필요한 시나리오.

| 단계 | 작업 | 설명 |
|------|------|------|
| 1 | CLI/라이브러리 API 정비 | create, read, modify, convert 등 공개 API 설계 |
| 2 | 빈 문서 생성 기능 | 템플릿 없이 새 HWP 문서를 코드로 생성 |
| 3 | MCP 프로토콜 연동 | JSON-RPC over stdio, 도구 정의 |
| 4 | 템플릿 기반 문서 생성 | 공문서/보고서 양식 + 매니페스트 지원 |

## 배포 전략

### 1. MCP 서버 (네이티브 바이너리)

AI Agent(Claude Code, Cowork 등)에서 직접 연동하는 주력 배포 형태.

```
Claude Code / Cowork
    │  MCP (JSON-RPC over stdio)
    ▼
rhwp-mcp (단일 바이너리)
    │
    ▼
  .hwp 파일 출력
```

### 2. PyPI 패키지 (Python 바인딩)

**PyO3**를 사용하여 Rust 코드를 Python 바인딩으로 빌드, `pip install rhwp`로 설치.

```python
import rhwp

# HTML → HWP 변환 (기존 html_to_pdf() 패턴과 동일)
rhwp.html_to_hwp("<h1>보고서</h1><p>내용...</p>", "report.hwp")

# 구조화된 데이터 → HWP 생성
doc = rhwp.create()
doc.add_paragraph("제목", style="heading1")
doc.add_table(data, headers=["항목", "금액"])
doc.save("output.hwp")

# 기존 HWP 읽기/수정
doc = rhwp.open("existing.hwp")
text = doc.extract_text()
doc.save("modified.hwp")
```

**기존 Claude 스킬과의 통합**:
```
현재:  데이터 → HTML 생성 → html_to_pdf()  → PDF 파일
추가:  데이터 → HTML 생성 → html_to_hwp() → HWP 파일
```

### 3. npm 패키지 (WASM)

Node.js 및 브라우저 환경용 WASM 패키지, `npm install rhwp`로 설치.

```javascript
import { htmlToHwp, createHwp } from 'rhwp';

// HTML → HWP 변환
const hwpBuffer = await htmlToHwp('<h1>보고서</h1>');
fs.writeFileSync('report.hwp', hwpBuffer);
```

### 4. Rust Crate

Rust 생태계 직접 활용, `cargo add rhwp`로 의존성 추가.

### SaaSpocalypse — AI 에이전트가 촉발한 SaaS 주가 대폭락 (2026년 2월)

2026년 2월 3일, Anthropic이 업무 자동화 도구 **Claude Cowork**를 공개하면서
SaaS 소프트웨어 기업 주가가 일제히 폭락했다. 시장은 이를 **"SaaSpocalypse"**로 명명했다.

**시장 충격 규모**:

| 지표 | 수치 |
|------|------|
| 나스닥 종합지수 | -1.43% |
| S&P 500 | -0.84% |
| 소프트웨어·서비스 섹터 | -3.8% |
| IGV 소프트웨어 ETF | -4.59% (1일) |
| 소프트웨어 시가총액 손실 | 단일 거래일 ~$300B, 7거래일 ~$1T |

**개별 기업 주가 낙폭** (2026년 연초 대비):

| 기업 | 하락률 | 업종 |
|------|--------|------|
| Figma | -40% | 디자인 협업 |
| HubSpot | -39% | 마케팅/CRM |
| Atlassian | -35% | 프로젝트 관리 |
| Intuit | -34% (1일 -11%) | 재무/세무 |
| Shopify | -29% | 이커머스 |
| ServiceNow | -28% (1일 -7%) | IT 서비스 관리 |
| Salesforce | -26% (1일 -7%) | CRM |
| Box | -17% | 클라우드 스토리지 |
| Thomson Reuters | -16% (1일) | 법률/금융 정보 |

**폭락의 본질 — "좌석 기반 과금 모델"의 붕괴**:

SaaS 기업의 성장 공식은 **"고용 증가 = 라이선스 판매 증가"**였다.
AI 에이전트가 업무를 자동화하면 기업은 더 적은 인원으로 동일한 업무를 처리할 수 있으므로,
소프트웨어 좌석(Seat) 수요 자체가 구조적으로 감소한다.

```
기존 공식:  직원 100명 → SaaS 100 Seat 구매 → 매출 성장
AI 시대:    직원 30명 + AI Agent → SaaS 30 Seat → 매출 70% 감소
```

이는 한컴의 딜레마와 정확히 동일한 구조이다:
- 한컴오피스 라이선스: **사용자 수 × 단가** → AI가 문서를 만들면 사용자 수 감소
- webhwp SaaS: **동시접속 수 × 구독료** → AI Agent는 세션을 짧게 쓰고 해제
- 한컴의 수익은 **"사람이 소프트웨어를 사용하는 횟수"**에 비례하는데, AI가 사람을 대체하면 그 기반이 소멸

**rhwp의 위치**: SaaSpocalypse는 "AI가 사람 대신 일하면 소프트웨어 좌석이 줄어든다"는 공포이다.
rhwp는 반대로 **"AI가 직접 사용하는 도구"**로 설계되었으므로, AI 확산이 곧 수요 증가로 이어진다.

### 한컴의 구조적 딜레마 — SaaSpocalypse가 증명한 현실

2026년 2월 SaaSpocalypse는 "좌석 기반 과금 모델의 종말"을 시장이 선언한 사건이다.
한컴은 이 구조적 전환의 정중앙에 위치한다.

**한컴의 3중 락인 구조**:

```
수익 모델 → 아키텍처 → 과금 방식 → 락인

  한컴오피스 라이선스 → 사용자 수 × 단가 → 좌석(Seat) 과금
  webhwp SaaS → 동시접속 수 × 구독료 → 세션 과금
  API 비공개 → 한컴 서비스 경유 필수 → 벤더 락인
```

이 구조는 SaaSpocalypse에서 폭락한 기업들과 정확히 동일한 패턴이다:

| SaaS 기업 | 과금 모델 | AI 시대 위협 | 한컴 대응 |
|-----------|----------|-------------|----------|
| Salesforce (-26%) | 사용자당 CRM 좌석 | AI가 영업 업무 대체 → 좌석 감소 | 한컴오피스 라이선스도 동일 구조 |
| Atlassian (-35%) | 팀원당 프로젝트 관리 | AI가 PM 업무 대체 → 좌석 감소 | webhwp 동시접속 과금도 동일 |
| HubSpot (-39%) | 좌석 기반 마케팅 도구 | AI가 마케팅 자동화 → 좌석 감소 | 문서 작성자 감소 → 라이선스 감소 |

**한컴이 AI 대응을 하려면 자사 수익을 파괴해야 한다**:

| 한컴의 AI 대응 | 수익 모델에 미치는 영향 | SaaSpocalypse 유사 사례 |
|----------------|----------------------|----------------------|
| 서버 의존성 제거 | SaaS 과금 근거 소멸 | Box (-17%): 클라우드 저장소 무의미화 |
| API 공개 | 한컴오피스 없이 HWP 생성 → 라이선스 감소 | ServiceNow (-28%): 자동화로 좌석 불필요 |
| 로컬 실행 허용 | 서버 접속 불필요 → 구독 모델 약화 | Intuit (-34%): AI가 세무 대체 |
| OT 우회 배치 모드 | webhwp 아키텍처 근본 변경 필요 | 기존 제품과 병행 불가 |

이는 전형적인 **혁신자의 딜레마**이다:

```
기존 사업 보호 → AI 대응 지연 → 시장 잠식
AI 대응 추진 → 기존 수익 훼손 → 주가 하락 (SaaSpocalypse 패턴)
```

시장은 이미 판단을 내렸다. 7거래일 만에 소프트웨어 섹터에서 **~$1T**가 증발했다.
"AI가 사람을 대체하면 소프트웨어 좌석이 줄어든다"는 것은 더 이상 가설이 아니라
**주가에 반영된 시장 컨센서스**이다.

**rhwp의 구조적 면역**:

rhwp는 SaaSpocalypse의 영향을 받지 않는다. 오히려 반대다:

| SaaS 기업 | rhwp |
|-----------|------|
| AI 확산 → 좌석 수요 감소 → 매출 하락 | AI 확산 → HWP 생성 수요 증가 → 사용량 증가 |
| 사용자 수 비례 과금 | 오픈소스, 과금 구조 없음 |
| 서버 비용이 사용량에 비례 | 로컬 처리, 서버 비용 0 |
| 기존 아키텍처가 AI 대응을 방해 | 처음부터 AI Agent용으로 설계 |

**AI가 많아질수록 rhwp의 가치는 올라가고, 한컴의 좌석은 줄어든다.**
