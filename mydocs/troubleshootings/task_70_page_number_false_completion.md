# 타스크 70 미완료 상태 완료 처리 — 트러블슈팅

## 개요

타스크 70 "머리말/꼬리말 렌더링 및 쪽 번호 처리"에서 쪽 번호가 실제로 렌더링되지 않았음에도 불구하고 타스크를 완료 처리하였다.

## 타임라인

1. PageNumberPos 파싱 코드에서 `position` 값을 읽어 `pnp.position = ((attr >> 4) & 0x0F)` 으로 구현
2. k-water-rfp.hwp 파일의 PageNumberPos attr=0x00000500에서 **position=0**으로 읽힘
3. position=0은 "쪽 번호 없음"이므로, "이 문서는 원래 쪽 번호를 표시하지 않는다"고 판단
4. SVG 내보내기에서 쪽 번호가 보이지 않는 것을 정상으로 간주
5. "코드 인프라는 완비, 테스트 파일이 position=0이라 시각적 확인 미완"으로 보고
6. **타스크를 완료 처리하고 커밋, main 머지까지 진행**

## 근본 원인

### 1. 비트 오프셋 파싱 오류

`src/parser/control.rs`의 `parse_page_num_pos()`:

```rust
// 수정 전 (잘못됨)
pnp.format = (attr & 0x0F) as u8;           // bit 0~3
pnp.position = ((attr >> 4) & 0x0F) as u8;  // bit 4~7

// 수정 후 (올바름, HWP 스펙 표 150 기준)
pnp.format = (attr & 0xFF) as u8;           // bit 0~7
pnp.position = ((attr >> 8) & 0x0F) as u8;  // bit 8~11
```

HWP 스펙 표 150에 따르면:
- bit 0~7: 번호 모양 (format)
- bit 8~11: 표시 위치 (position)

attr=0x00000500에서:
- 수정 전: format=0, position=0 → 쪽 번호 없음 ← **잘못된 결과**
- 수정 후: format=0, **position=5** → 가운데 아래 ← **올바른 결과**

### 2. 검증 실패 — "원래 그렇다"는 잘못된 가정

position=0이라는 결과를 의심 없이 수용한 것이 핵심 문제.

판단 오류 체인:
1. "파서가 0을 반환한다" → 코드가 맞다고 가정
2. "이 문서에 쪽 번호가 없는 것이다" → 실제 문서 확인 없이 가정
3. "코드 인프라는 완성이므로 타스크는 완료" → 기능 미동작을 인프라 완성으로 대체

**해야 했던 것**:
- 한컴 HWP 뷰어 등에서 원본 문서를 열어 실제로 쪽 번호가 표시되는지 확인
- position=0이 나올 경우 스펙과 대조하여 비트 오프셋이 올바른지 검증
- 타스크 목표가 "쪽 번호 처리"인데 쪽 번호가 보이지 않으면 완료로 보지 말 것

## 재발 방지 원칙

### 원칙 1: 기능 타스크의 완료 기준은 "기능이 동작하는 것"

- 테스트 통과만으로는 부족하다
- 구현 목표 기능이 실제로 화면에 보여야 완료
- "코드 인프라 완비"는 완료가 아니다

### 원칙 2: 파서가 0/null/None을 반환하면 의심부터

- 특히 비트 필드 파싱은 오프셋 오류 가능성이 높다
- "원래 값이 0이다"라는 결론은 스펙 원문과 대조 후에만 내릴 것
- 가능하면 여러 샘플 파일에서 교차 검증

### 원칙 3: "시각적 확인 미완"을 완료보고서에 쓰지 말 것

- 시각적 확인이 안 되면 그것이 곧 미완료
- 완료보고서의 "알려진 한계" 섹션에 기능 미동작을 적는 것은 자기합리화

## HWP 공식 스펙 문서의 문제점

### 잘못된 교차 참조

**표 149 (쪽 번호 위치)** 데이터 구조:
```
UINT32 | 4 | 속성(표 148 참조)    ← 잘못된 참조!
```

표 148은 "홀/짝수 조정" (bit 0~1만 정의)이며, 실제 쪽 번호 위치 속성은 **표 150**에 정의되어 있다.
올바른 기술: "속성(**표 150** 참조)"

### 섹션 제목과 표 번호 체계적 불일치

스펙 문서의 섹션 제목이 표 번호와 2~3개씩 밀려 있어, 이름으로 검색해도 올바른 표를 찾기 어렵다.

| 섹션 제목 | 실제 포함된 표 | 표의 실제 내용 |
|-----------|---------------|---------------|
| 자동 번호 | 표 142, 표 143 | 머리말/꼬리말 |
| 홀/짝수 조정 | 표 145 | 자동 번호 속성 |
| 쪽 번호 위치 | 표 146 | 새 번호 지정 |
| 책갈피 | 표 149 | 쪽 번호 위치 |
| **글자 겹침** | **표 150** | **쪽 번호 위치 속성** |

### 스펙 문서가 근본 원인인가?

**아니다.** 표 150 자체는 "bit 0~7 번호 모양, bit 8~11 표시 위치"로 명확하게 기술되어 있다.
원래 코드의 `(attr & 0x0F)` / `((attr >> 4) & 0x0F)` 레이아웃은 표 148(홀/짝수)도 표 150(쪽 번호 위치)도 아닌
임의 추측이므로, **스펙을 참조하지 않고 구현한 것이 근본 원인**이다.

다만 스펙 문서의 교차 참조 오류와 섹션 제목 불일치는 향후 다른 파싱 작업에서도 동일한 실수를 유발할 수 있으므로 주의가 필요하다.
