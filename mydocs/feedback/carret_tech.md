**캐럿(Caret/Cursor) 처리**는 겉보기엔 단순해 보여도 조판 엔진의 논리적 무결성을 증명하는 가장 정교한 지점입니다. 특히 단순 텍스트 편집기가 아닌, 장평/자간이 실시간으로 변하는 DTP 수준의 워드프로세서라면 더욱 그렇습니다.

2026년의 기술적 성숙도와  Rust/AI 역량을 고려할 때, **'좌표 역산(Inverse Mapping)'**과 **'가상 캐럿 엔티티'** 관점에서 이 문제를 해결하는 설계 전략을 제안합니다.

---

## 1. 공간 인덱싱을 통한 Hit Testing (Quad-tree/R-tree)

사용자가 화면의 특정 지점을 클릭하거나 키보드를 입력할 때, "이 좌표가 논리적으로 어떤 글자의 앞인가, 뒤인가?"를 찾는 과정입니다.

* **문제:** 장평과 자간이 적용된 글자들은 고정 너비가 아닙니다.
* **해결:** 레이아웃 엔진이 모든 글리프(Glyph)의 최종 위치를 확정할 때, 각 글자의 **Bounding Box** 정보를 공간 인덱싱 트리(예: `rstar` 크레이트)에 저장합니다.
* **효과:** 마우스 클릭 시 $O(\log n)$의 속도로 해당 위치의 글자 인덱스를 찾아낼 수 있으며, 이는 rust wasm 방식의 성능과 맞물려 매우 쾌적한 반응성을 제공합니다.

## 2. 가상 캐럿(Virtual Caret)과 렌더링 동기화

캐럿을 시스템 커서가 아닌, 씬 그래프 상의 **독립적인 엔티티(Entity)**로 관리해야 합니다.

* **Logical Positioning:** 캐럿은 "몇 번째 문단의 몇 번째 글자 뒤"라는 논리적 주소를 가집니다.
* **Visual Transformation:** 레이아웃 엔진은 해당 글자의 장평(Scale X)과 자간(Offset)이 적용된 **최종 Baseline 좌표**를 계산하여 캐럿 엔티티의 전역 변환 행렬(Global Transform)에 넘깁니다.
* **결과:** 글자가 가로로 늘어나면(장평) 캐럿도 그에 맞춰 정밀하게 이동하며, 소수점 단위의 조판 결과물 위에서도 캐럿이 글자 사이를 정확히 파고듭니다.

## 3. IME 조합 중 레이아웃(Composition-aware Reflow)

한글은 초/중/종성이 합쳐지는 과정에서 글자의 너비가 실시간으로 변합니다.

* **상태 분리:** 입력 중인 '조합 문자'를 문서 구조(DOM)에 완전히 삽입하기 전, 레이아웃 엔진의 **임시 버퍼(Sandbox)**에서 미리 배치해 봅니다.
* **실시간 피드백:** 조합 중인 글자 때문에 줄바꿈이 일어날지 AI와 함께 설계한 가변 플로우 알고리즘이 미리 예측하여, 사용자가 타이핑하는 순간 문단 전체가 자연스럽게 밀려나는 'Live Reflow'를 구현합니다.

## 4. 2026년형 설계: 데이터 중심(Data-Oriented) 캐럿 시스템

Rust의 병렬성을 활용한다면, 캐럿의 위치를 찾는 'Hit Test'를 GPU 가속 없이도 CPU 단에서 수천 개의 문단에 대해 병렬로 수행할 수 있습니다.

| 단계 | 처리 내용 | 적용 기술 |
| --- | --- | --- |
| **Input Event** | 키보드/마우스 이벤트 수신 | SDL2 / Web Event |
| **State Update** | 논리적 커서 위치(Cursor Index) 갱신 | Rust Atomic/Immutable State |
| **Layout Sync** | 변경된 인덱스의 물리 좌표() 계산 | Knuth-Plass 기반 Solver |
| **Scene Commit** | ThorVG 씬 그래프에 캐럿 위치 업데이트 | ThorVG `Matrix` 변환 |
| **Draw** | 최종 렌더링 | ThorVG + SDL2/WASM |
