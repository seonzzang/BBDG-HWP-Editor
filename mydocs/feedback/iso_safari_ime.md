google docs 는 ios 사파리에서 한글 IME 가 잘되는군요. 일반적인 경우 자모가 분리되어 입력되는데 말이죠.

# A1

에드워드님, 역시 엔진을 직접 만드시는 분이라 가장 까다로운 **'IME(Input Method Editor) 지옥'**의 핵심을 바로 짚어내셨군요.

iOS 사파리에서 한글 자모가 분리되는 소위 **'천지인 현상'**이나 **'자모 분리(Ghost characters)'** 문제는 웹 개발자들에게는 악명 높은 난제입니다. 구글 독스가 이를 매끄럽게 처리하는 것은 표준 브라우저의 `contenteditable` 기능을 믿지 않고 **독자적인 '입력 프록시(Input Proxy)' 레이어**를 구축했기 때문입니다.

기술적인 메커니즘을 분석해 보고서 형태로 정리해 드립니다.

---

## [기술 분석] Google Docs의 iOS 사파리 한글 IME 처리 메커니즘

### 1. 왜 일반적인 웹 에디터는 자모가 분리되는가?
iOS 사파리의 `contenteditable` 영역은 한글과 같은 **조합형 문자(Composition)**를 처리할 때 매우 불안정합니다.
* **이벤트 타이밍 이슈:** 브라우저가 자음과 모음이 합쳐지는 도중에 `input` 이벤트를 성급하게 발생시키거나, 가상 키보드와의 통신 과정에서 조합 상태(Composition State)를 놓치기 때문입니다.
* **결과:** "가나다"를 입력할 때 브라우저는 `ㄱ`, `ㅏ`, `ㄴ`을 각각 독립적인 문자로 인식하여 캔버스나 DOM에 뿌려버리고, 조합이 완료된 시점을 놓쳐 "ㄱㅏㄴㅏㄷㅏ"가 남게 됩니다.

### 2. Google Docs의 해결책: "Shadow Input & Composition Events"
구글 독스는 사용자가 화면(Canvas)을 보고 있지만, 실제 입력은 **화면에 보이지 않는 투명한 `textarea`**에서 일어나게 합니다.

* **Invisible Input (프록시 레이어):** 사용자가 캔버스의 특정 위치를 클릭하면, 구글 독스는 그 좌표 바로 위에 1x1 픽셀 크기의 투명한 `textarea`를 배치하고 포커스를 줍니다.
* **조합 이벤트 제어:** 브라우저의 기본 입력을 차단하는 대신, 다음 세 가지 이벤트를 정밀하게 가로챕니다.
    1.  **`compositionstart`**: 한글 입력이 시작됨을 감지.
    2.  **`compositionupdate`**: 자모가 합쳐지는 중간 과정(예: `ㄱ` -> `가`)을 실시간으로 읽어와 캔버스에 '임시'로 그림.
    3.  **`compositionend`**: 한 글자가 완성되었음을 알리고, 최종 데이터를 문서 모델에 반영.



### 3. iOS 사파리 전용 최적화 (The "Docs" Secret)
iOS 사파리에서는 특히 **'포커스 유지'**와 **'가상 키보드 높이'**가 문제입니다. 구글 독스는 이를 위해 다음과 같은 트릭을 씁니다.

* **Zero-latency Reflection:** 사용자가 투명 `textarea`에 한 글자를 입력하는 즉시, JS 엔진이 그 값을 읽어 캔버스에 렌더링합니다. 이때 **'조합 중'**임을 시각적으로 나타내기 위해 글자 아래에 점선이나 실선을 직접 그립니다.
* **Focus Guard:** iOS 사파리는 캔버스를 터치할 때 포커스를 잃기 쉽습니다. 구글 독스는 모든 터치 이벤트를 인터셉트하여 강제로 투명 `textarea`에 `focus()`를 유지시킵니다.
* **IME 버퍼 관리:** iOS 사파리가 간혹 `compositionend` 없이 다음 글자로 넘어가는 버그를 방지하기 위해, 자체적인 **타이머와 버퍼**를 두어 특정 시간 동안 입력이 없으면 강제로 조합을 완료시키는 로직을 가지고 있습니다.

### 4. 에드워드님의 Mjolnir(Rust/Wasm) 프로젝트를 위한 제언

에드워드님이 **Rust**와 **Wasm** 기반으로 에디터를 만드신다면, 이 구글 독스의 모델을 반드시 따라가야 합니다. 

1.  **Rust-JS Bridge:** `compositionupdate` 이벤트를 Rust 레이어로 초고속 전송해야 합니다. 자모 분리를 막으려면 '현재 조합 중인 문자'를 위한 별도의 **Temporary Buffer**를 Rust 구조체에 설계하세요.
2.  **Canvas Text Shaping:** 조합 중인 글자가 캔버스에 그려질 때, 옆의 글자들이 밀려나는 **레이아웃 연산(Line Breaking)**을 실시간(Real-time)으로 수행해야 합니다. (이 부분이 조판 엔진의 실력을 가르는 지점입니다.)
3.  **Careful Cursor Mapping:** 투명 `textarea`의 커서 위치와 캔버스의 시각적 커서 위치를 동기화하는 것이 가장 고통스러운 작업이 될 것입니다. 특히 **장평과 자간**이 적용된 경우, 픽셀 단위 좌표 계산이 정교해야 합니다.

**"구글 독스가 잘되는 이유는 사파리가 똑똑해서가 아니라, 구글 엔지니어들이 사파리의 멍청함을 수만 줄의 코드로 메꿨기 때문입니다."**

혹시 Rust에서 이 **`composition` 이벤트**를 효율적으로 수신하여 **SkParagraph** 레이아웃에 즉시 반영하는 구조(Architecture)를 같이 설계해 볼까요? 아니면 iOS 사파리에서 가상 키보드가 올라올 때 **Viewport 좌표계가 뒤틀리는 문제**를 해결하는 방법이 궁금하신가요?
