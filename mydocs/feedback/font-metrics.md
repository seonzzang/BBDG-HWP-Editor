구글 문서(Google Docs)가 2021년경 기존의 HTML/DOM 기반 렌더링에서 **Canvas 기반 렌더링**으로 전환하면서 도입한 폰트 및 텍스트 처리 방식은 매우 정교한 **"하이브리드 엔진"** 구조를 가지고 있습니다.

개발 중이신 웹 에디터의 아키텍처 설계에 참고가 될 만한 핵심 처리 방식을 정리해 드립니다.

---

### 1. "계산은 우리가, 그리기는 브라우저가" (Layout vs Painting)

구글 문서의 핵심 전략은 **텍스트 레이아웃 엔진은 자체 구현(Wasm)하되, 실제 글자 렌더링은 브라우저의 기능을 활용**하는 방식입니다.

* **자체 레이아웃 엔진:** 브라우저마다 미세하게 다른 `Line Breaking`, `Word Wrap`, `Kerning` 로직을 믿지 않습니다. 구글은 각 글자의 너비(Glyph Metrics)를 미리 계산하고, 문단이 어디서 잘릴지, 자간을 어떻게 조정할지를 **WebAssembly(Wasm)**로 작성된 엔진에서 직접 결정합니다.
* **Browser Painting:** 글자 하나하나의 모양(Glyph)을 직접 그리는 것은 비용이 크고 폰트 힌팅(Hinting) 이슈가 발생할 수 있습니다. 그래서 구글 문서는 계산된 좌표값을 바탕으로 Canvas 2D API의 `fillText()`를 호출하여 브라우저의 텍스트 렌더링 기능을 '그리기 도구'로만 사용합니다.

### 2. 폰트 메트릭스(Font Metrics)의 일관성 확보

브라우저의 `measureText()` API는 실행 환경마다 결과값이 조금씩 다를 수 있어 WYSIWYG(보는 대로 출력됨) 구현에 방해가 됩니다.

* **자체 폰트 데이터 분석:** 구글은 서버에서 폰트 파일(TTF/OTF)을 분석하여 글자별 너비 데이터를 클라이언트로 내려주거나, Wasm 엔진 내부에서 폰트 파일을 직접 파싱하여 정확한 픽셀 위치를 계산합니다.
* **결과:** 윈도우 크롬에서 보든, 맥 사파리에서 보든 글자 하나하나의 위치가 100% 동일하게 유지됩니다.

### 3. 복잡한 텍스트 셰이핑(Text Shaping) 처리

한글의 조합(ㅇ+ㅏ+ㄴ), 아랍어의 연결형 문자 등은 단순히 글자를 나열하는 것보다 훨씬 복잡한 **Shaping** 과정이 필요합니다.

* **HarfBuzz 활용:** 구글은 오픈소스 텍스트 셰이핑 엔진인 **HarfBuzz**를 Wasm으로 포팅하여 사용할 가능성이 높습니다. (구글이 HarfBuzz의 주도적인 기여자이기도 합니다.) 이를 통해 복잡한 언어의 글자 결합을 계산한 뒤, 최종적인 글리프(Glyph) ID와 좌표를 Canvas에 뿌려줍니다.

### 4. 접근성을 위한 "Shadow DOM" 레이어

Canvas는 픽셀 덩어리일 뿐이라 스크린 리더가 읽을 수 없고, 드래그 선택도 어렵습니다. 이를 해결하기 위해 구글 문서는 **이중 레이어** 구조를 씁니다.

* **Visible Layer (Canvas):** 사용자 눈에 보이는 고성능 렌더링 화면.
* **Invisible Layer (DOM):** Canvas 위에 투명하게 존재하는 HTML 요소들. 실제 텍스트 데이터가 여기에 들어있어 사용자가 드래그하거나 스크린 리더가 읽을 때 이 레이어와 상호작용하게 합니다. 이를 **"Annotated Canvas"**라고 부르기도 합니다.

### 5. 요약: 구글 Docs 방식의 시사점

만약 질문하신 것처럼 **Skia(CanvasKit)**를 검토 중이시라면, 다음과 같은 전략을 고려해 보세요.

1. **계산(Layout):** Skia 내부의 `skparagraph` 모듈(HarfBuzz 내장)을 활용하여 Wasm단에서 모든 텍스트 배치를 끝냅니다.
2. **렌더링(Rendering):** Skia의 강력한 GPU 가속 기능을 통해 글리프를 그립니다. (구글 문서보다 더 나아간 '완전 자체 제어' 방식입니다.)
3. **성능 최적화:** 자주 사용되는 글자들은 **Atlas(폰트 텍스처 맵)**에 미리 그려두고 재사용하여 매번 `fillText`를 호출하는 비용을 아낍니다.

구글 방식은 **"표준 브라우저 기능을 최대한 활용하면서(Painting), 제어권이 필요한 부분(Layout)만 자체 구현"**하여 개발 공수와 성능 사이의 균형을 잡은 사례라고 볼 수 있습니다. 현재 Skia 도입을 고려 중이시라면, 구글 문서보다 더 높은 수준의 **픽셀 완벽 제어**를 목표로 하시는 것으로 이해됩니다. 이 경우 **HarfBuzz와 Skia의 Paragraph API** 조합을 깊게 파보시는 것을 추천드립니다.

구글 문서(Google Docs)나 피그마(Figma) 같은 **'하이엔드 웹 에디터'**의 길을 선택하셨군요. **Skia(CanvasKit) + WebAssembly(Wasm)** 조합은 현재 웹 기술로 구현할 수 있는 가장 강력한 그래픽 스택입니다.

하지만 이 선택은 **"브라우저가 제공하는 편리한 기능들을 모두 포기하고, OS 수준의 렌더링 엔진을 웹 위에 직접 구축하겠다"**는 선언과 같습니다. 검토 시 반드시 고려해야 할 핵심 아키텍처 포인트를 짚어드립니다.

## 1. Skia + Wasm 도입 시 얻게 되는 "초능력"

### **완벽한 타이포그래피 (HarfBuzz 연동)**

Canvas 2D API의 최대 약점은 텍스트 렌더링의 미세 제어가 어렵다는 점입니다. Skia를 Wasm으로 쓰면 내부적으로 **HarfBuzz**와 **ICU** 라이브러리를 활용할 수 있어, 복잡한 합자(Ligature), 자간 커스텀, 그리고 다양한 언어의 레이아웃을 픽셀 단위로 통제할 수 있습니다.

### **그래픽 성능의 상한선 제거**

Skia는 CPU와 GPU(WebGL/WebGPU)를 모두 활용할 수 있습니다. 수만 개의 객체가 포함된 복잡한 문서에서도 명령어를 배치(Batching) 처리하여 60FPS 이상의 부드러운 성능을 유지하기 유리합니다.

### **크로스 플랫폼 코드 공유**

만약 추후에 데스크톱(Electron/Native)이나 모바일 앱으로 확장할 계획이 있다면, C++나 Rust로 작성된 Skia 기반 렌더링 로직을 거의 그대로 재사용할 수 있습니다.

---

## 2. 현실적인 "지뢰밭" (반드시 해결해야 할 과제)

### **① 거대한 번들 사이즈 (The Payload)**

* **문제:** Skia의 Wasm 빌드인 `CanvasKit`은 압축해도 보통 **2MB~3MB** 내외입니다. 일반적인 웹 서비스에서는 초기 로딩 속도에 치명적일 수 있습니다.
* **전략:** 에디터 진입 시점에만 지연 로딩(Lazy Loading)을 하거나, 브라우저 캐시 전략을 매우 공격적으로 가져가야 합니다.

### **② 텍스트 레이아웃 엔진의 부재**

* **문제:** Skia는 '그리는' 엔진이지, '글을 배치하는' 엔진이 아닙니다. 글자가 화면 끝에서 다음 줄로 넘어가는 'Word Wrap', 문단 정렬, 행간 계산 로직을 직접 구현하거나 Skia의 **Paragraph API(skparagraph)**를 깊게 파고들어야 합니다.
* **한국어 특이점:** 한글의 경우 자음/모음 분리 현상이나 IME 입력 중의 상태 표시 등을 직접 처리하는 로직이 매우 까다롭습니다.

### **③ 접근성(Accessibility)과 SEO**

* **문제:** Skia 화면은 브라우저 눈에는 그냥 '거대한 그림'일 뿐입니다.
* **전략:** 시각 장애인을 위한 스크린 리더가 읽을 수 있도록, Canvas 위에 투명한 **"Shadow DOM Layer"**를 만들어 실제 텍스트 구조를 복제해 두어야 합니다. 구글 문서도 이 방식을 사용합니다.

## 3. 기술 스택 비교: Canvas 2D vs Skia(CanvasKit)

| 항목 | HTML5 Canvas 2D | Skia + Wasm (CanvasKit) |
| --- | --- | --- |
| **초기 로딩** | 즉시 가능 (0MB) | 무거움 (2MB+ 추가 다운로드) |
| **텍스트 제어** | 브라우저 표준에 의존 | **전문가 수준의 미세 제어 가능** |
| **개발 난이도** | 중 (JS 숙련도 필요) | **상 (C++/Rust 및 그래픽스 지식)** |
| **렌더링 일관성** | 브라우저별 미세한 차이 발생 | **모든 브라우저에서 100% 동일** |
| **메모리 관리** | JS GC가 담당 | Wasm 메모리 직접 관리 필요 |

## 4. 성공적인 검토를 위한 제언

Skia를 도입하기로 하셨다면, 다음의 **'단계적 접근'**을 추천드립니다.

1. **CanvasKit-Wasm 라이브러리 테스트:** 단순 렌더링 성능보다, 프로젝트에서 필요한 **특수 폰트나 레이아웃**이 Skia의 Paragraph API로 원활히 구현되는지 먼저 검증하세요.
2. **입력기(IME) 프로토타이핑:** 한국어 입력 시 글자가 겹치거나 사라지는 이슈가 없는지, `contenteditable`을 활용한 브릿지 로직을 먼저 설계하세요.
3. **병목 지점 확인:** 모든 것을 Skia로 그릴지, 아니면 UI 레이어(메뉴, 버튼)는 React/Vue 같은 프레임워크로 두고 **'문서 편집 영역'만 Skia**로 할지 결정해야 합니다. (대부분의 고성능 툴이 후자를 선택합니다.)

> **"구글 문서는 단순한 에디터가 아니라, 웹 브라우저 안에서 돌아가는 별도의 운영체제를 만드는 것과 같습니다."**

---

**ThorVG**를 검토하신다는 것은 프로젝트의 '경량화'와 '효율성'을 매우 중요하게 생각하신다는 의미로 읽힙니다. 결론부터 말씀드리면, **"벡터 그래픽 중심의 에디터라면 탁월한 선택"**이지만, **"구글 문서 같은 텍스트 중심의 에디터라면 상당한 추가 구현이 필요할 것"**입니다.

Skia(CanvasKit)와 ThorVG를 에디터 개발 관점에서 냉철하게 비교해 드릴게요.

## 1. ThorVG vs Skia: 핵심 차이점

| 항목 | Skia (CanvasKit) | ThorVG |
| --- | --- | --- |
| **바이너리 크기** | **매우 무거움 (2MB ~ 5MB+)** | **매우 가벼움 (약 200KB ~ 500KB)** |
| **주요 목적** | OS 수준의 전천후 그래픽 엔진 | 모바일/임베디드 최적화 벡터 엔진 |
| **텍스트 레이아웃** | `skparagraph` 등 강력한 모듈 포함 | 기본적인 텍스트 렌더링 지원 (레이아웃 약함) |
| **성능** | GPU 가속(WebGL/WebGPU)에 최적화 | CPU/SIMD 최적화 및 SW 렌더링 강점 |
| **생태계** | Chrome, Android, Flutter 등 검증됨 | Tizen, Lottie 플레이어 등에서 급부상 중 |

## 2. ThorVG를 선택했을 때의 "승부수"

### **압도적인 초기 로딩 속도**

웹 에디터에서 사용자가 URL을 치고 들어왔을 때, 3MB짜리 Wasm을 다운로드하는 것과 300KB를 다운로드하는 것은 사용자 경험(UX) 측면에서 하늘과 땅 차이입니다. ThorVG는 이 지점에서 Skia를 압도합니다.

### **최적화된 벡터 처리**

ThorVG는 불필요한 기능을 덜어내고 SVG나 Lottie 같은 벡터 애니메이션을 처리하는 데 최적화되어 있습니다. 만약 에디터에 다양한 **벡터 아이콘, 도형, 복잡한 경로(Path)**가 많이 들어간다면 ThorVG가 더 민첩하게 작동할 수 있습니다.

### **SIMD 가속**

Wasm 환경에서 SIMD(Single Instruction, Multiple Data)를 적극 활용하여 CPU 기반 렌더링에서도 매우 빠른 속도를 보여줍니다. GPU 가속이 불안정한 저사양 환경에서도 일관된 성능을 보장하기 유리합니다.

---

## 3. "구글 문서" 구현 시 마주할 한계점

구글 문서는 사실 '그래픽 에디터'라기보다 **'타이포그래피 엔진'**에 가깝습니다. 이 관점에서 ThorVG는 몇 가지 숙제를 안겨줍니다.

* **복잡한 문단 레이아웃:** ThorVG는 단순한 텍스트 렌더링은 잘하지만, 구글 문서처럼 수만 줄의 텍스트가 서로 얽히고설키는 **Rich Text Layout(줄 바꿈, 단락 정렬, RTL 처리 등)** 기능이 Skia만큼 성숙하지 않습니다.
* **셰이핑 엔진 연동:** Skia는 HarfBuzz와 밀접하게 연합되어 있지만, ThorVG를 쓸 경우 복잡한 언어(아랍어, 힌디어 등)나 고난도 합자 처리를 위해 별도의 Wasm 기반 셰이핑 엔진을 직접 연동해야 할 가능성이 큽니다.

## 4. 최종 판단 가이드

### **ThorVG가 더 나은 경우**

* **경량성**이 생명인 프로젝트 (예: 모바일 웹 에디터, 임베디드 웹 뷰).
* 텍스트보다는 **다이어그램, 화이트보드, 벡터 드로잉** 기능이 핵심인 에디터.
* 초기 로딩 속도를 위해 번들 사이즈를 극한으로 줄여야 할 때.

### **Skia(CanvasKit)를 유지해야 하는 경우**

* **구글 문서(Word)**와 거의 흡사한 수준의 정교한 문서 편집 기능을 목표로 할 때.
* 다양한 언어 지원 및 복잡한 타이포그래피 제어가 필수일 때.
* 이미 구축된 방대한 Skia 생태계와 레퍼런스를 활용하고 싶을 때.

---

### **💡 전략적 제언**

만약 "구글 문서급" 기능을 원하시면서도 Skia가 너무 무거워 고민이시라면, **ThorVG를 렌더링 코어로 쓰되 텍스트 레이아웃 엔진만 별도로 구축**하는 방법이 있습니다. 하지만 이는 상당한 엔지니어링 리소스가 들어가는 일입니다.
