# Hyper-Waterfall: AI 시대의 소프트웨어 개발 방법론

> AI가 없던 시대에는 불가능했던 방법론.
> 워터폴의 규율과 애자일의 속도를 동시에.

## 1. 왜 새로운 방법론이 필요한가

### 워터폴의 실패

```
요구분석 ──→ 설계 ──→ 구현 ──→ 테스트 ──→ 배포
(2주)       (2주)    (4주)    (2주)      (1주)
```

워터폴은 올바른 철학을 가지고 있었다 — 계획하고, 문서화하고, 검증하라. 하지만 **속도가 너무 느렸다.** 계획서를 쓰는 데 2주, 구현하는 데 4주. 시장은 기다려주지 않았다.

### 애자일의 타협

```
[스프린트1] → [스프린트2] → [스프린트3] → ...
   2주          2주          2주
```

그래서 애자일이 등장했다. "동작하는 소프트웨어가 포괄적인 문서보다 우선한다." 속도를 얻었지만 대가가 있었다:

- **문서가 사라졌다** — "코드가 곧 문서"라는 변명
- **방향이 흔들렸다** — 스프린트마다 우선순위가 바뀜
- **기술 부채가 쌓였다** — "나중에 리팩토링하자"가 영원히 미뤄짐
- **지식이 사람 머리에 갇혔다** — 핵심 개발자가 떠나면 프로젝트가 멈춤

### 바이브 코딩의 함정

AI가 등장하자 더 극단적인 시도가 나왔다:

```
"AI야 만들어줘" → [수락] → [수락] → [수락] → ... → 💥
```

계획도 없고, 문서도 없고, 검증도 없다. AI가 준 코드를 읽지도 않고 수락한다. 겉보기에는 빠르지만, **이해하지 못한 코드는 고칠 수도 없다.**

### 근본 문제

기존 방법론의 딜레마:

```
규율 (문서, 계획, 검증)  ←──── 트레이드오프 ────→  속도 (빠른 구현, 빠른 배포)
        워터폴 ◄────────────────────────────────► 애자일/바이브코딩
```

워터폴은 규율은 있지만 느리다. 애자일은 빠르지만 규율이 약하다. **둘 다 가질 수는 없었다** — AI가 없던 시대에는.

## 2. Hyper-Waterfall 정의

### 핵심 원리

**거시적으로는 워터폴, 미시적으로는 애자일 — AI가 이 둘을 동시에 가능하게 한다.**

```
거시 (프로젝트 수준) — 워터폴의 규율:
  계획 ──→ 설계 ──→ 구현 ──→ 검증 ──→ 배포
  │         │        │        │        │
  ▼         ▼        ▼        ▼        ▼
  문서화    문서화    문서화    문서화    문서화

미시 (타스크 수준, 수 시간) — 애자일의 속도:
  구현 → 테스트 → 피드백 → 수정 → 테스트 → ... (빠른 반복)
  │       │         │        │       │
  AI     자동화    사람 판단   AI    자동화
```

거시적 방향은 **워터폴의 규율**로 통제한다 — 계획서, 승인, 단계별 보고, 최종 검증이 있다. 미시적 실행은 **애자일의 빠른 반복**으로 수행한다 — AI와 즉각적인 피드백 루프를 돌린다. 이 전체 사이클이 **몇 시간** 안에 완료된다.

### 워터폴과의 차이

전통 워터폴에서 느렸던 이유는 **사람이 모든 것을 했기 때문**이다:

- 사람이 계획서를 쓰고 (2일)
- 사람이 코드를 작성하고 (1주)
- 사람이 테스트를 만들고 (3일)
- 사람이 문서를 정리하고 (2일)

Hyper-Waterfall에서는:

- **AI가 계획서를 쓰고** → 사람이 검토/승인 (10분)
- **AI가 코드를 작성하고** → 사람이 방향 제시 (1시간)
- **AI가 테스트를 만들고** → CI가 자동 검증 (2분)
- **AI가 문서를 정리하고** → 사람이 확인 (5분)

**규율은 그대로, 속도만 100배.**

### 애자일과의 차이

| | 애자일 | Hyper-Waterfall |
|--|--------|-----------------|
| 계획 | 백로그 카드 (한 줄) | 구현 계획서 (페이지 단위) |
| 문서 | 선택사항 | 필수 (AI가 생성) |
| 방향 | 스프린트마다 조정 | 마일스톤으로 고정 |
| 리뷰 | 코드 리뷰 (선택) | 계획 승인 + 결과 검증 (필수) |
| 속도 | 2주 스프린트 | 수 시간 사이클 |
| 지식 관리 | 사람 머리 속 | 문서화 (AI가 기록) |
| 기술 부채 | 누적 | 매 사이클에서 해결 |

## 3. 핵심 원칙

Hyper-Waterfall의 세 가지 원칙. 이것을 지키지 않으면 바이브 코딩으로 전락한다.

### 제 1원칙: 구현 가치(목표)가 AI 컨텍스트에 유지되도록 한다

AI는 현재 대화의 컨텍스트 안에서만 일관성을 유지한다. 프로젝트의 비전, 아키텍처 원칙, 품질 기준이 AI의 컨텍스트에서 벗어나면, AI는 방향을 잃고 단편적인 코드만 생산한다.

- **CLAUDE.md**에 프로젝트 규칙, 빌드 방법, 워크플로우를 명시
- **타스크 등록 시** 목표와 범위를 명확히 기술
- **컨텍스트가 길어지면** 핵심 맥락을 요약하여 재주입

### 제 2원칙: 작업 통제권을 작업지시자가 유지해야만 한다

AI에게 방향을 맡기는 순간 프로젝트는 표류한다. 무엇을 만들지, 어떤 순서로 할지, 언제 멈출지 — 이 모든 결정은 사람이 한다.

- **계획은 AI가 제안하되, 승인은 사람이 한다**
- **AI가 임의로 작업 종료를 제안하지 않는다** — 작업지시자가 결정
- **아키텍처 변경은 반드시 사람의 판단을 거친다**
- **"좋은 것 같습니다"로 넘어가지 않는다** — 직접 검증한다

### 제 3원칙: 주기적으로 AI 컨텍스트 유지를 확인해야 한다

AI의 적은 **메모리와 토큰 사용량**이다. 대화가 길어지면 초반의 맥락을 잃고, 같은 실수를 반복하거나 이전에 합의한 방향과 다른 코드를 생산한다.

- **컨텍스트 압축 시점**을 인지한다 — AI가 이전 결정을 잊기 시작하면 경고 신호
- **핵심 결정사항을 문서로 외부화**한다 — AI 메모리가 아닌 파일에 기록
- **새 세션 시작 시** 이전 맥락을 요약하여 전달한다
- **메모리 시스템**을 활용한다 — 프로젝트별 영구 기억 저장

> 이 세 원칙은 "사람은 절대 생각을 멈추지 않는다"의 구체적 실천이다.

## 4. 역할 분담

### 작업지시자 (사람)

사람은 **생각하는 역할**에 집중한다:

- 방향 설정: "다음에 뭘 해야 하는가?"
- 우선순위: "무엇이 더 중요한가?"
- 품질 판단: "이것이 충분히 좋은가?"
- 아키텍처 결정: "이 구조가 올바른가?"
- 도메인 지식: "한컴은 이 경우 어떻게 동작하는가?"
- 피드백: "이 부분이 잘못되었다, 왜냐하면..."

### AI 페어 프로그래머

AI는 **실행하는 역할**에 집중한다:

- 분석: 코드베이스 탐색, 원인 추적
- 계획: 구현 계획서 작성
- 구현: 코드 작성, 테스트 생성
- 문서: 보고서, 기술 문서, 커밋 메시지
- 디버깅: 로그 분석, 수정안 제시
- 반복: 피드백 반영, 재시도

### 핵심 원칙

> **사람은 절대 생각을 멈추지 않는다.**

AI가 아무리 뛰어나도, 방향을 결정하고 품질을 판단하는 것은 사람이다. AI의 출력을 읽지 않고 수락하는 순간, Hyper-Waterfall은 바이브 코딩으로 전락한다.

## 5. 프로세스

### 타스크 사이클

```
1. 타스크 등록
   └─ 작업지시자: GitHub Issue 생성, 범위 정의
   
2. 수행 계획서
   └─ AI: 계획서 작성 (최소 3단계, 최대 6단계)
   └─ 작업지시자: 검토 → 승인 또는 수정 요청

3. 단계별 구현
   └─ AI: 코드 작성 + 테스트
   └─ AI: 단계 완료 보고서 작성
   └─ 작업지시자: 검증 → 승인 또는 피드백

4. 피드백 반영
   └─ 작업지시자: 피드백 문서 작성 (mydocs/feedback/)
   └─ AI: 피드백 반영, 수정

5. 최종 보고
   └─ AI: 최종 결과 보고서 작성
   └─ 작업지시자: 승인 → 이슈 클로즈
```

### 문서 구조

매 타스크가 만들어내는 산출물:

```
mydocs/
├── orders/yyyymmdd.md          ← 오늘 할일 (타스크 목록 + 상태)
├── plans/task_{N}.md           ← 수행 계획서
├── plans/task_{N}_impl.md      ← 구현 계획서
├── working/task_{N}_step{M}.md ← 단계별 완료 보고서
├── working/task_{N}_final.md   ← 최종 결과 보고서
└── feedback/                   ← 코드 리뷰 피드백
```

이 문서들은 부담이 아니다 — **AI가 작성하기 때문이다.** 사람은 검토하고 승인할 뿐이다.

### 품질 관문

모든 타스크는 다음을 통과해야 한다:

1. **계획 승인** — 구현 전에 방향이 맞는지 확인
2. **테스트 통과** — cargo test (783+), cargo clippy
3. **시각적 검증** — SVG 렌더링 결과를 눈으로 확인
4. **E2E 검증** — 브라우저 렌더링까지 확인
5. **회귀 방지** — 기존 샘플 파일의 페이지 수 변동 없음

## 6. 왜 AI가 있어야 가능한가

### 문서화 비용의 소멸

전통 워터폴에서 개발자가 가장 싫어했던 것 — 문서 작성. "코드 한 줄 쓰는 시간에 문서 한 페이지를 써야 한다"는 불만이 애자일을 만들었다.

Hyper-Waterfall에서는 AI가 문서를 작성한다. 사람은 검토만 한다. rhwp 프로젝트의 724개 문서 중 사람이 직접 작성한 것은 피드백 문서뿐이다.

### 실행 속도의 혁명

```
전통 워터폴 1사이클:  2~4주
애자일 1스프린트:     2주
Hyper-Waterfall 1사이클: 2~4시간
```

같은 규율을 유지하면서 100배 빠르다. 이것이 가능한 이유는 AI가 계획, 구현, 테스트, 문서를 **동시에** 수행하기 때문이다.

### 지식의 외부화

애자일에서 가장 큰 리스크는 "버스 팩터" — 핵심 개발자가 버스에 치이면 프로젝트가 멈추는 것. 지식이 사람 머리에만 있기 때문이다.

Hyper-Waterfall에서는 모든 결정, 모든 디버깅 과정, 모든 아키텍처 선택이 문서화된다. AI가 자동으로 기록하기 때문이다. 개발자가 떠나도 문서가 남는다.

## 8. 방법론 비교 요약

```
         규율 높음
            ↑

            │         ● 바이브 코딩

```

## 9. 적용 조건

Hyper-Waterfall은 누구나 쓸 수 있는 것이 아니다.

### 필요 조건

1. **도메인 전문성** — AI에게 올바른 방향을 제시하려면 도메인을 알아야 한다
2. **아키텍처 판단력** — AI의 설계 제안을 평가할 수 있어야 한다
3. **품질에 대한 기준** — "이 정도면 충분하다"를 판단할 수 있어야 한다
4. **AI 도구 숙련도** — AI의 강점과 한계를 알아야 한다
5. **규율** — 편하다고 바이브 코딩으로 빠지지 않는 자기 통제

### 적합한 프로젝트

- 복잡한 도메인 (역공학, 렌더링, 파싱 등)
- 높은 품질 기준 (상용 소프트웨어 수준)
- 문서화가 중요한 프로젝트
- 1인 또는 소규모 팀

### 부적합한 경우

- 도메인 지식이 없는 상태에서 AI에 의존
- "빨리 만들고 버리자"식 프로토타입
- AI 출력을 검증할 능력이 없는 경우

## 10. 결론

> AI는 배율기다. 프로세스가 없으면 빠른 혼돈을, 좋은 프로세스가 있으면 비범한 결과를 만든다.

Hyper-Waterfall은 AI 시대에 **잃어버린 규율을 되찾는 방법론**이다.

애자일이 속도를 위해 포기한 것들 — 문서, 계획, 체계적 검증 — 을 AI의 힘으로 되찾는다. 워터폴이 규율을 위해 포기한 것 — 속도 — 을 AI의 힘으로 되찾는다.

둘 다 포기할 필요가 없었다. **AI가 있으니까.**
