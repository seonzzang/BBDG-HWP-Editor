# AI 페어프로그래밍 가이드

## 개요

이 문서는 rhwp 프로젝트에서 AI(Claude)와 사람(작업지시자)이 함께 복잡한 소프트웨어를 만드는 과정에서 축적된 페어프로그래밍 패턴과 원칙을 정리한 것입니다.

Rust + WASM + Canvas + E2E + 역공학이라는 높은 난이도의 프로젝트에서, 작업지시자의 도메인 지식과 AI의 구현력이 결합되어 3일 만에 167개 파일, 6,500줄의 코드를 생산하고 5개 이슈를 하루에 해결하는 속도를 달성했습니다.

## 역할 분담

### 작업지시자 (사람)

- **방향 설정**: 무엇을 해야 하는지, 왜 해야 하는지를 결정
- **도메인 지식**: 한컴 HWP의 조판 규칙, 격자 비교, 시각적 품질 판단
- **품질 기준**: "우회방식은 허용하지 않겠습니다", "한컴처럼 해야 합니다"
- **즉각적 피드백**: 스크린샷, 격자 캡처, "표가 위에 있습니다" 같은 시각적 교정
- **작업 시간 관리**: 작업의 시작과 종료를 결정

### AI (Claude)

- **구현**: 코드 작성, 빌드, 테스트 실행
- **분석**: 코드 구조 탐색, 버그 원인 추적, 수치 계산
- **설계**: API 설계, 자료구조 설계, 테스트 프레임워크 구축
- **문서화**: 매뉴얼, 커밋 메시지, 보고서 작성
- **검증**: 네이티브 테스트, E2E 테스트, regression 확인

## 핵심 원칙

### 1. 시각적 피드백으로 즉시 교정한다

```
작업지시자: [스크린샷 첨부] "표가 베이스라인보다 위에 있습니다"
   → AI: 수치 분석 → 원인 파악 → 코드 수정 → SVG 재생성
작업지시자: [스크린샷 확인] "이제 맞습니다" 또는 "아직 2px 높습니다"
   → AI: 추가 보정 → 재확인
```

한컴의 격자보기(1mm 간격)를 활용한 정밀 비교가 대표적입니다. 작업지시자가 격자 캡처를 제공하면 AI가 px 단위로 분석하여 정확한 조판 규칙을 도출합니다.

**실제 사례**: 인라인 TAC 표의 세로 정렬
- 1차: `table_y = baseline - table_h` → "표가 약간 높습니다"
- 2차: `table_y = y` (줄 시작) → "더 높아졌습니다"
- 3차: `table_y = baseline + om_bottom - table_h` → "한컴과 일치합니다"

### 2. 우회하지 않고 정면 돌파한다

작업지시자가 우회 방식을 거부하면, 근본적인 해결책을 찾아야 합니다.

**실제 사례**: 인라인 표 뒤에서 텍스트 입력
- 우회: 텍스트를 먼저 모두 입력한 후 표를 삽입 → **거부됨**
- 정면 돌파: 논리적 오프셋 체계 설계 → `insert_text_at` 수정 → `navigateNextEditable` 활용 → 한컴 방식 입력 흐름 구현

### 3. 검증 가능한 구조를 만든다

수동 확인에 의존하면 regression을 잡을 수 없습니다.

```
수동 확인 → "스크린샷이 맞아 보입니다"
   ↓ 발전
자동 검증 → 선언적 시나리오 → 렌더 트리 좌표 측정 → 규칙 기반 판정
```

**발전 과정**:
1. SVG 내보내기로 눈으로 확인
2. E2E 스크린샷 캡처 (기록용)
3. 렌더 트리 좌표 추출 (`getPageRenderTree`)
4. 규칙 검증기 (`inline-order`, `table-baseline-align`, `stable-after-enter`)
5. 선언적 시나리오 실행기 (JSON으로 시나리오 정의 → 자동 실행 + 검증)

### 4. 작업지시자가 방향을 잡고 AI가 실행한다

AI가 스스로 작업 범위를 정하거나 종료를 제안하지 않습니다.

```
작업지시자: "이번 타스크에 포함시켜주세요"  → AI: 범위 확장
작업지시자: "별도 이슈로 등록하세요"        → AI: 분리
작업지시자: "우회방식은 허용하지 않겠습니다" → AI: 근본 해결
작업지시자: "한컴처럼 해야 합니다"          → AI: 기준 상향
```

### 5. 간과한 것을 발견하면 즉시 반영한다

**실제 사례**: 페이지네이션 전파 범위

```
작업지시자: "14페이지까지 진행하면 다음 페이지는 변경될 일이 없어야 합니다"
   → AI: 분석 → 14페이지의 188.9px 여유 공간 발견
   → 수렴 감지 프레임워크 설계 → 증분 측정 캐시 최적화
```

작업지시자의 한 마디가 설계 방향을 근본적으로 바꿀 수 있습니다.

## 효과적인 커뮤니케이션 패턴

### 시각적 증거 기반 대화

| 비효율적 | 효과적 |
|---------|--------|
| "표 위치가 이상합니다" | [격자 캡처] "표 하단이 베이스라인보다 1mm 위입니다" |
| "렌더링이 틀립니다" | [한컴 스크린샷 + 우리 스크린샷] "셀 텍스트와 호스트 텍스트의 베이스라인이 다릅니다" |
| "페이지네이션이 안 됩니다" | "13페이지에서 엔터를 입력하면 14페이지 표가 15페이지로 밀립니다" |

### 점진적 정밀화

```
1차: "인라인 표가 텍스트 사이에 들어가야 합니다"     ← 방향
2차: [격자 캡처] "표 하단과 텍스트 베이스라인 정렬"   ← 규칙
3차: "어센더/디센더가 포함된 텍스트로 다시 확인"      ← 엣지 케이스
4차: "outer_margin_bottom만큼 더 내려야 합니다"      ← 정밀 보정
```

### 실패를 통한 학습

```
AI: baseline - table_h 방식 적용 → SVG 출력
작업지시자: "시각적으로 변화가 없습니다. 4px 정도 내려야 합니다"
AI: line_height 방식 → 6.3px 이동 → "너무 많이 내려갔습니다"
AI: baseline + outer_margin_bottom → 3.77px → 한컴과 일치
```

실패→피드백→수정 사이클이 빠르게 반복되어 정확한 규칙을 도출합니다.

## 디버깅 워크플로우

### 레이아웃 버그

```bash
# 1. 시각적 식별
rhwp export-svg sample.hwp --debug-overlay -o output/

# 2. 페이지 배치 확인
rhwp dump-pages sample.hwp -p 23

# 3. 문단 속성 조사
rhwp dump sample.hwp -s 0 -p 239

# 4. HWPX↔HWP 비교 (파싱 차이)
rhwp ir-diff sample.hwpx sample.hwp
```

### 페이지네이션 버그

```rust
// 네이티브 테스트로 재현
let mut doc = HwpDocument::from_bytes(&bytes).unwrap();
doc.convert_to_editable_native().unwrap();
doc.paginate();

let pages_before = doc.pagination[0].pages.len();
doc.split_paragraph_native(0, 199, 0).unwrap();
let pages_after = doc.pagination[0].pages.len();

// 저장 후 재로드와 비교
let exported = doc.export_hwp_native().unwrap();
let mut doc2 = HwpDocument::from_bytes(&exported).unwrap();
// → 편집 결과와 재로드 결과가 일치해야 함
```

### E2E 조판 버그

```javascript
// 선언적 시나리오로 재현
const scenario = {
  name: '버그 재현',
  steps: [
    { type: 'text', value: '텍스트' },
    { type: 'inlineTable', rows: 2, cols: 2, ... },
    { type: 'enter' },  // ← 여기서 버그 발생
  ],
};
// → 각 step마다 스크린샷 + 렌더 트리 스냅샷 자동 수집
```

## 프로젝트 구조가 AI 페어프로그래밍에 미치는 영향

### 잘 동작하는 구조

- **CLI 도구 체인**: `export-svg`, `dump-pages`, `dump`, `ir-diff` — AI가 즉시 실행하여 결과 확인
- **네이티브 테스트**: Rust `#[test]` — 빠른 피드백 (7초에 783개)
- **E2E 프레임워크**: CDP 기반 — 시각적 결과를 프로그래밍으로 검증
- **선언적 시나리오**: JSON으로 시나리오 정의 → 재현 가능
- **명확한 브랜치 전략**: `local/task{N}` → `local/devel` → `devel` → `main`

### AI가 어려워하는 것

- **시각적 판단**: "한컴과 비슷해 보이나요?" — 작업지시자의 눈이 필수
- **도메인 규칙 추론**: "표 하단 = 베이스라인 + outer_margin_bottom" — 한컴 역공학 데이터 필요
- **우선순위 결정**: "이번 타스크에 포함할까요?" — 작업지시자가 결정
- **작업 종료 시점**: 작업지시자만 결정

## 체크리스트: AI와 효과적으로 작업하기

### 작업 시작 전

- [ ] GitHub Issue 등록 (타스크 번호 = Issue 번호)
- [ ] 브랜치 생성 (`local/task{N}`)
- [ ] 재현 가능한 샘플 준비 (HWP 파일 또는 시나리오 JSON)

### 작업 중

- [ ] 시각적 피드백은 스크린샷/격자 캡처로 제공
- [ ] "이것은 맞습니다" / "이것은 틀립니다"를 명확히
- [ ] 우회가 아닌 근본 해결을 요구
- [ ] 중간 결과를 자주 확인 (SVG, E2E, 테스트)

### 작업 완료

- [ ] 전체 테스트 통과 확인 (`cargo test`)
- [ ] E2E 검증 시나리오 추가
- [ ] 커밋 + Issue close
- [ ] `devel` → `origin/devel` push
- [ ] 오늘 할일 문서 갱신

## 결론

AI 페어프로그래밍의 핵심은 **역할 분담의 명확성**입니다. 사람은 방향·판단·피드백을, AI는 구현·분석·검증을 담당합니다. 이 분담이 명확할수록 작업 속도와 품질이 동시에 높아집니다.

이 프로젝트는 단순한 HWP 뷰어가 아니라, AI와 사람이 함께 복잡한 소프트웨어를 만드는 과정 자체가 레퍼런스입니다.
