# 타스크 33: 언어별 폰트 분기 (한영 혼합 문서 폰트 정확도)

## 배경

HWP 파일은 `CharShape.font_ids[7]`에 7개 언어 카테고리별 폰트 ID를 저장한다:

| 인덱스 | 언어 |
|--------|------|
| 0 | 한국어 |
| 1 | 영어 |
| 2 | 한자 |
| 3 | 일본어 |
| 4 | 기타 |
| 5 | 기호 |
| 6 | 사용자 |

현재 rhwp는 `style_resolver.rs`에서 `font_ids[0]` (한국어)만 사용하여 모든 문자를 렌더링한다. 따라서 "안녕 Hello"와 같은 한영 혼합 텍스트에서 영문도 한글 폰트(예: 함초롬돋움)로 렌더링되는 문제가 있다.

webhwp(한컴 웹 에디터)는 문자마다 해당 언어 카테고리의 폰트를 개별 적용한다.

## 목표

- 텍스트의 각 문자에 대해 Unicode 범위 기반으로 언어 카테고리를 판별
- 해당 카테고리의 `font_ids[lang]` → `font_faces[lang][font_id]`로 정확한 폰트 이름 조회
- 폰트가 바뀌는 지점에서 TextRun을 분할하여 각 구간에 올바른 폰트 적용
- 기존 테스트 전체 통과 + 신규 테스트 추가

## 범위

### 포함

- Unicode 범위 → 언어 카테고리 매핑 함수
- 스타일 해석 시 언어별 폰트 분기
- TextRun 분할 (동일 CharShape 내에서 폰트가 다른 구간)
- 텍스트 레이아웃 JSON에 정확한 폰트 이름 반영
- 서식 툴바의 폰트 표시 정확도 (캐럿 위치 기준)

### 제외

- 문자 단위 개별 `fillText` 렌더링 (성능 트레이드오프, 별도 타스크)
- 텍스트 외곽선/그림자 효과 (별도 타스크)
- 취소선/밑줄 렌더링 (별도 타스크)

## 기술 접근

1. **언어 판별**: Unicode 코드포인트 범위로 한국어/영어/한자/일본어/기호 분류
2. **스타일 해석 확장**: `resolve_char_style()`에서 `font_ids[lang]` 조회하되, 동일 CharShape 내에서 폰트 전환이 필요한 경우 별도 처리
3. **Run 분할**: 텍스트 레이아웃 단계에서 폰트가 바뀌는 지점을 감지하여 Run을 분할
4. **JSON 확장**: `getPageTextLayout()` JSON의 각 Run에 정확한 `fontFamily` 반영

## 리스크

- Run 분할 증가로 인한 렌더링 성능 영향 → 연속 동일 폰트 구간은 병합
- 테이블 셀 내부 텍스트도 동일하게 적용 필요
- 기존 `measureText` 기반 너비 계산이 폰트별로 정확해질 수 있음 (긍정적 영향)

## 예상 변경 파일

| 파일 | 작업 |
|------|------|
| `src/renderer/style_resolver.rs` | 언어 판별 함수 + 언어별 폰트 조회 |
| `src/renderer/layout.rs` | Run 분할 로직 |
| `src/wasm_api.rs` | JSON 출력에 정확한 fontFamily 반영 |
