## 수행 내용

### Section 4: 미구현 기능 우선순위 (P0/P1/P2)

등급 C (신규 구현 필요) 약 290개 API를 마이그레이션 필수도에 따라 4단계로 분류하였다.

| 등급 | 수 | 핵심 항목 |
|------|-----|----------|
| P0 (필수) | 7개 영역 | 필드 시스템(13 API), 커서/위치(96 API), Action/ParameterSet(19 API), Undo/Redo(2), 검색/치환(9), 텍스트 스캔(4), 선택/블록(36+) |
| P1 (중요) | 10개 영역 | 이미지 삽입, 머리말/꼬리말, 페이지 설정, 셀 서식, 각주/미주, 문서 삽입, 하이퍼링크, 컨트롤 관리, 배포용 문서, 이벤트 |
| P2 (선택) | 7개 영역 | 그리기 도구, 문단번호, 문자표, 단 정의, 달력 변환, 표 크기 조절, 문서 정보 |
| X (스텁) | 3개 영역 | UI 제어, 맞춤법, 엔진 설정 |

### Section 5: 마이그레이션 시나리오별 필요 API

공공기관 활용 패턴을 4가지 시나리오로 분류하고 실제 코드 예시와 함께 커버리지를 분석하였다.

| 시나리오 | 필요 API | 현재 커버리지 | 핵심 블로커 |
|----------|---------|-------------|-----------|
| 문서 뷰어 | 5개 | **100%** | 없음 (즉시 마이그레이션 가능) |
| 기안문 자동 생성 | 12개 | **25%** | 필드 시스템 |
| 양식 편집 | 25개 | **32%** | 커서/필드/Action 시스템 |
| 완전 편집기 | 449개 | **27%** | 전체 P0+P1+P2 |

**핵심 발견**: 가장 빈번한 시나리오(기안문 자동 생성)의 블로커가 **필드 시스템 단일 영역**이므로, 이를 최우선 구현하면 최대 효과를 얻을 수 있다.

### Section 6: rhwp 고유 강점

웹기안기 대비 rhwp의 차별화 포인트를 아키텍처 강점(5항목)과 기능 강점(6항목)으로 정리하였다.

- **서버 비의존**: WASM 로컬 처리로 인프라 비용 절감
- **오픈소스**: 벤더 종속 탈피
- **AI 통합 최적화**: 좌표 기반 API가 AI 함수 호출에 적합
- **다중 렌더링**: SVG/HTML/Canvas 동시 지원
- **전략적 포지셔닝**: "사람이 UI로 편집" (한컴) vs "AI가 API로 제어" (rhwp)

### Section 7: 호환 레이어 구현 로드맵

4개 Phase 점진적 구현 로드맵을 수립하였다.

| Phase | 목표 | 전체 커버리지 |
|-------|------|-------------|
| Phase 1 | 기안문 자동 생성 100% | ~30% |
| Phase 2 | 기본 편집기 60% | ~50% |
| Phase 3 | 고급 편집기 90% | ~75% |
| Phase 4 | 완전 호환 95% | ~95% |

Phase 1에 대해 구체적 구현 계획(JS 호환 레이어 프레임워크 + Rust WASM 필드 API 시그니처)을 제시하였다.
