# HWP/HWPX 링크 드래그앤드롭 요구사항

## 1. 목적

브라우저에서 검색되거나 다운로드 가능한 `HWP` / `HWPX` 링크를 BBDG HWP Editor 창으로 드래그앤드롭했을 때,
사용자가 별도 다운로드 과정을 거치지 않아도 문서를 열 수 있게 한다.

이 기능은 기존의 로컬 파일 드래그앤드롭을 대체하지 않으며, 그 위에 `링크 기반 문서 열기`를 추가하는 1차 확장 기능이다.

## 2. 범위

### 포함

- 브라우저에서 앱 창으로 드래그된 데이터 중 아래 포맷 분석
  - `File`
  - `text/uri-list`
  - `text/plain`
  - `DownloadURL`
- HTTP/HTTPS 기반 링크 지원
- 직접 파일 URL(`.hwp`, `.hwpx`) 다운로드 후 자동 열기
- 확장자가 없어도 응답 헤더를 기반으로 HWP/HWPX 파일 여부 판별
- 임시 폴더 저장 후 기존 문서 로딩 경로로 열기
- 실패 시 사용자 안내 메시지 표시

### 제외

- HTML 페이지를 파싱해서 다운로드 링크를 다시 추출하는 기능
- 로그인/세션이 필요한 사설 사이트 다운로드 지원
- 브라우저 확장 프로그램 연동
- 다중 파일 일괄 다운로드/일괄 열기
- 백그라운드 다운로드 관리자

## 3. 사용자 시나리오

### 시나리오 A: 직접 파일 링크 드롭

1. 사용자가 브라우저에서 `.hwp` 또는 `.hwpx` 직접 링크를 끌어온다.
2. 앱은 링크를 감지한다.
3. 앱은 임시 폴더로 파일을 다운로드한다.
4. 기존 파일 열기 경로를 통해 문서를 연다.

### 시나리오 B: 확장자 없는 다운로드 링크 드롭

1. 사용자가 브라우저에서 다운로드 링크를 끌어온다.
2. 앱은 URL 확장자만으로는 판단하지 못한다.
3. 앱은 응답 헤더를 조회한다.
4. `Content-Type` / `Content-Disposition` 기준으로 HWP/HWPX 파일 여부를 판단한다.
5. 파일로 확인되면 다운로드 후 문서를 연다.

### 시나리오 C: 지원하지 않는 링크 드롭

1. 사용자가 일반 웹페이지 링크나 검색 결과 링크를 드롭한다.
2. 앱은 파일 링크가 아님을 판단한다.
3. 앱은 문서를 열지 않고 사용자에게 안내한다.

## 4. 기능 요구사항

### FR-1. 드래그 데이터 분석

앱은 드롭 이벤트 발생 시 다음 우선순위로 데이터를 해석해야 한다.

1. `File` 객체
2. `DownloadURL`
3. `text/uri-list`
4. `text/plain`

### FR-2. 기존 로컬 파일 드롭 우선 유지

드롭된 데이터에 실제 `File` 객체가 포함되어 있으면, 기존 로컬 파일 열기 흐름을 그대로 사용해야 한다.

### FR-3. 링크 스킴 제한

앱은 `http://`, `https://` 링크만 처리해야 한다.

아래는 지원하지 않는다.

- `blob:`
- `data:`
- `file:`
- `javascript:`
- 그 외 사용자 정의 스킴

### FR-4. 직접 파일 링크 판별

링크 경로가 `.hwp` 또는 `.hwpx`로 끝나면 직접 파일 링크로 간주하고 다운로드를 시도해야 한다.

### FR-5. 응답 헤더 기반 파일 판별

링크가 직접 파일 확장자를 포함하지 않더라도, 아래 조건 중 하나를 만족하면 HWP/HWPX 파일로 간주할 수 있어야 한다.

- `Content-Type`이 HWP/HWPX 관련 MIME 타입인 경우
- `Content-Disposition`의 파일명이 `.hwp` 또는 `.hwpx`인 경우
- `Content-Type`이 `application/octet-stream`이어도 `Content-Disposition` 파일명이 HWP/HWPX인 경우

### FR-6. 임시 파일 저장

다운로드된 파일은 앱 전용 임시 디렉터리에 저장해야 한다.

- 문서명은 원본 파일명 또는 안전한 대체 이름 사용
- 기존 문서를 덮어쓰지 않도록 충돌 방지
- 파일 열기 이후에도 필요한 동안은 유지

### FR-7. 자동 열기

다운로드가 완료되면 기존 문서 열기 경로와 동일한 검증/로딩 플로우를 사용해 문서를 열어야 한다.

### FR-8. 사용자 피드백

다운로드 및 열기 과정에서 사용자에게 상태를 안내해야 한다.

예:

- `링크에서 문서를 다운로드하는 중...`
- `문서를 여는 중...`
- `지원되지 않는 링크입니다.`
- `다운로드에 실패했습니다.`

### FR-9. 실패 시 안전 종료

다운로드 또는 판별 실패 시:

- 현재 열린 문서는 유지되어야 한다.
- 편집기 본체는 정상 상태를 유지해야 한다.
- 중간 생성된 불완전 파일은 정리되어야 한다.

## 5. 비기능 요구사항

### NFR-1. 안정성

이 기능은 기존 로컬 파일 드롭/열기 기능을 깨뜨리면 안 된다.

### NFR-2. 보안

외부 링크 처리 시 아래를 강제한다.

- HTTP/HTTPS만 허용
- 로컬 파일 스킴 차단
- 파일명 sanitize
- 임시 폴더 경로 고정

### NFR-3. 성능

헤더 판별과 다운로드는 UI를 장시간 멈추게 하지 않아야 한다.

### NFR-4. 추적 가능성

초기 구현에는 진단 로그를 남겨야 한다.

예:

- 드롭 데이터 포맷
- 최종 판별된 URL
- 헤더 판별 결과
- 다운로드 경로
- 실패 원인

## 6. 지원 우선순위

### 1차 목표

- `File` / `DownloadURL` / `text/uri-list` / `text/plain` 분석
- 직접 파일 URL 다운로드
- 헤더 기반 HWP/HWPX 판별
- 다운로드 후 자동 열기

### 2차 후보

- `[파일] -> [URL로 열기]`
- 실패 원인별 세분화 안내
- 다운로드 캐시/최근 링크 기록

### 보류

- HTML 페이지 파싱
- 브라우저 확장 연동
- 인증 세션 다운로드

## 7. 승인 기준

아래 조건을 모두 만족하면 1차 구현 완료로 본다.

1. 로컬 파일 드래그앤드롭이 기존과 동일하게 열린다.
2. `.hwp` 또는 `.hwpx` 직접 링크 드롭 시 다운로드 후 문서가 열린다.
3. 확장자 없는 링크라도 헤더 판별로 HWP/HWPX가 확인되면 열린다.
4. 일반 웹페이지 링크 드롭 시 문서는 열리지 않고 안내만 표시된다.
5. 실패 시 편집기 본체는 깨지지 않는다.
